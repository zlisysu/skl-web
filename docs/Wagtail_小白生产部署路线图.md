## 1. 理解本地电脑、Linux 服务器、浏览器三者的关系

### 这一章要解决什么问题

先不要急着部署，也不要急着看 Nginx、Gunicorn、PostgreSQL。

第一步只解决一个最基础的问题：

```text
我现在本地能访问网站，和以后别人通过服务器访问网站，到底有什么区别？
```

如果这个问题没搞清楚，后面讲 IP、端口、SSH、Nginx、域名时都会混在一起。

### 当前本地开发是什么样

你现在在本地项目目录运行：

```bash
.venv/bin/python manage.py runserver
```

然后浏览器访问：

```text
http://127.0.0.1:8000
```

这条链路是：

```text
你本地电脑上的浏览器
  -> 你本地电脑自己的 8000 端口
    -> Django runserver
      -> lab-web 项目代码
        -> 本地 db.sqlite3
        -> 本地 media/
```

这里最关键的是：

```text
这个网站现在是跑在你自己的电脑上。
```

### `127.0.0.1` 是什么意思

`127.0.0.1` 是一个特殊 IP，意思是：

```text
当前这台机器自己。
```

所以你在自己电脑浏览器里访问：

```text
http://127.0.0.1:8000
```

意思是：

```text
访问我自己这台电脑的 8000 端口。
```

如果你在 Linux 服务器里面访问：

```text
http://127.0.0.1:8000
```

那意思就变成：

```text
访问 Linux 服务器自己的 8000 端口。
```

所以要特别注意：

| 你在哪里访问 `127.0.0.1` | 它指的是谁 |
| --- | --- |
| 在你本地电脑浏览器里 | 你本地电脑自己 |
| 在 Linux 服务器终端里 | Linux 服务器自己 |
| 在别人电脑浏览器里 | 别人电脑自己 |

结论：

```text
127.0.0.1 不是公网地址，别人不能通过你的 127.0.0.1 访问你的网站。
```

### IP 和端口分别是什么

可以先用一个很粗糙但好理解的比喻：

| 概念 | 类比 |
| --- | --- |
| IP | 一栋楼的地址 |
| 端口 | 这栋楼里的某个门 |
| 服务 | 门后面正在接待的人 |

比如：

```text
127.0.0.1:8000
```

可以理解成：

```text
当前这台机器的 8000 号门。
```

再比如 Linux 服务器公网 IP 是：

```text
1.2.3.4
```

那么：

```text
http://1.2.3.4:8000
```

可以理解成：

```text
访问 1.2.3.4 这台服务器的 8000 号门。
```

### 浏览器、服务器、Django 的关系

浏览器只是发请求的工具。

服务器是长期运行网站的机器。

Django/Wagtail 是生成网页内容的程序。

本地开发时：

```text
浏览器
  -> 本地电脑
    -> Django runserver
```

正式部署时：

```text
浏览器
  -> Linux 服务器
    -> Nginx
      -> Gunicorn
        -> Django / Wagtail
```

现在先不用理解 Nginx 和 Gunicorn 的细节，只要知道正式部署时网站不是跑在你本地电脑，而是跑在 Linux 服务器上。

### 为什么要有 Linux 服务器

你本地电脑不适合当正式网站服务器，原因很简单：

- 你的电脑可能会关机。
- 你的电脑网络环境不一定有公网 IP。
- 你的电脑不适合长期暴露给外部访问。
- 你的电脑上的 `runserver` 是开发用的，不是生产用的。

Linux 服务器的作用是：

- 24 小时开机。
- 有固定或相对稳定的公网 IP。
- 能被外部浏览器访问。
- 能长期运行 Nginx、Gunicorn、数据库等服务。

### 部署最终要改变什么

现在是：

```text
你本地浏览器
  -> 你本地电脑的 127.0.0.1:8000
    -> 本地 Django runserver
```

部署后要变成：

```text
别人浏览器
  -> Linux 服务器 IP 或域名
    -> 服务器上的网站服务
```

所以部署不是让别人访问你本地电脑上的 `runserver`。

部署是：

```text
把项目放到 Linux 服务器上
  -> 在服务器上运行网站
    -> 让别人通过服务器 IP 或域名访问它
```

### 这一章的检查问题

学完这一章，你应该能回答：

1. `127.0.0.1` 是不是一个别人能访问的公网地址？
2. 你本地运行 `runserver` 时，网站是跑在本地电脑还是服务器？
3. IP 和端口大概分别表示什么？
4. 部署的目标是让网站跑在你本地电脑，还是跑在 Linux 服务器？

### 这一章的结论

这一章只要记住四句话：

1. 本地电脑负责开发，Linux 服务器负责正式运行网站。
2. `127.0.0.1` 永远表示当前这台机器自己。
3. IP 是机器地址，端口是这台机器上的入口。
4. 部署就是把网站从“只在自己电脑能访问”，变成“别人能通过服务器 IP 或域名访问”。

## 2. 确认能用 SSH 登录 Linux 服务器

### 这一章要解决什么问题

这一步只解决一个问题：

```text
你有没有办法真正进入那台 Linux 服务器去操作它？
```

如果不能登录服务器，后面所有部署步骤都无从谈起，因为：

- 你没法安装环境
- 你没法上传代码
- 你没法启动 Gunicorn
- 你没法配置 Nginx

### SSH 是什么

SSH 可以先理解成：

```text
在你自己电脑上，远程打开服务器的终端。
```

你敲的命令不再是在本地电脑执行，而是在服务器上执行。

所以 SSH 登录成功以后，你操作的就是：

```text
那台 Linux 服务器
```

不是你手边这台电脑。

### 为什么部署前一定先学会 SSH

后面的部署动作，基本都发生在服务器里，比如：

- 创建项目目录
- 安装 Python、Node、Nginx
- 拉取 Git 仓库
- 运行 `gunicorn`
- 配置 Nginx
- 检查日志

这些都不是在浏览器里完成的，而是在 SSH 终端里完成的。

### SSH 登录长什么样

最常见的登录命令是：

```bash
ssh 用户名@服务器IP
```

例如：

```bash
ssh root@1.2.3.4
```

或者：

```bash
ssh ubuntu@1.2.3.4
```

这里三部分分别表示：

| 部分 | 意思 |
| --- | --- |
| `ssh` | 使用 SSH 工具发起远程登录 |
| `用户名` | 你要以谁的身份登录服务器 |
| `服务器IP` | 你要登录哪台服务器 |

### 常见登录方式

常见有两种：

1. 密码登录

   你输入 `ssh 用户名@服务器IP` 后，服务器要求输入密码。

2. SSH key 登录

   你的本地电脑里保存一个私钥，服务器保存对应公钥。登录时不一定需要手输密码。

对小白来说，先只需要知道：

```text
无论是密码登录还是 key 登录，目标都是先成功进到服务器终端里。
```

### 怎么判断“我真的已经登录到服务器了”

登录成功后，终端一般会发生这些变化：

- 提示符变化了
- 当前用户名可能变成 `root` 或 `ubuntu`
- 当前目录变成服务器上的某个目录

最简单的检查命令有三个：

```bash
whoami
pwd
hostname
```

它们分别用来查看：

| 命令 | 看什么 |
| --- | --- |
| `whoami` | 当前登录用户是谁 |
| `pwd` | 当前所在目录是哪里 |
| `hostname` | 当前机器叫什么名字 |

如果你在 SSH 里运行这些命令，看到的是服务器信息，不是本地电脑信息。

### 一个最容易混淆的点

很多人第一次用 SSH 会混淆：

```text
我现在这条命令是在本地执行，还是在服务器执行？
```

你可以这样分辨：

- 还没执行 `ssh ...` 之前：命令在本地电脑执行
- 执行 `ssh ...` 并成功登录之后：命令在服务器执行

所以：

```bash
ssh ubuntu@1.2.3.4
```

这条命令本身是在本地发起的。

但登录成功后你再执行：

```bash
pwd
```

就是在服务器上执行的。

### Windows / WSL 下怎么理解

你现在项目在 Windows 路径和 WSL/bash 环境里工作。

这不影响 SSH 的核心概念。你只要记住：

- 你的终端可以在本地打开 bash
- 在本地 bash 里执行 `ssh ...`
- 登录成功后，这个终端就连到了服务器

所以“我看到的还是一个黑窗口”不重要，关键是：

```text
这个黑窗口现在连的是本地，还是服务器。
```

### 这一章最基础的实操目标

这一步不要求你部署网站，只要求你做到：

1. 知道服务器的 IP
2. 知道登录用户名
3. 能执行 `ssh 用户名@服务器IP`
4. 登录后能运行 `whoami`、`pwd`、`hostname`
5. 知道自己此时已经在服务器终端里

### 这一章的检查问题

学完这一章，你应该能回答：

1. SSH 是干什么用的？
2. `ssh 用户名@服务器IP` 这条命令每部分是什么意思？
3. 登录成功后，你后面敲的命令是在本地执行还是在服务器执行？
4. `whoami`、`pwd`、`hostname` 分别能帮你确认什么？

### 这一章的结论

这一章只要记住三句话：

1. SSH 是远程进入 Linux 服务器终端的方式。
2. 登录成功后，你执行的大多数部署命令都是在服务器上跑的。
3. 部署前第一件事不是配 Nginx，而是先确保你真的能登录并操作服务器。

## 3. 确认服务器公网 IP、端口和防火墙

### 这一章要解决什么问题

这一步要解决的是：

```text
就算服务器上已经跑了一个网站，为什么浏览器还是可能访问不到？
```

答案通常和这三件事有关：

- 服务器有没有公网 IP
- 服务监听的是哪个端口
- 防火墙或安全组有没有放行对应端口

### 什么是公网 IP

公网 IP 可以理解成：

```text
互联网里别人能找到你这台服务器的地址。
```

如果服务器公网 IP 是：

```text
1.2.3.4
```

那么外部浏览器理论上可以访问：

```text
http://1.2.3.4
```

或者：

```text
http://1.2.3.4:8000
```

但“理论上可以访问”不代表一定能成功，因为还要看端口和防火墙。

### 什么是端口

一台服务器不止能跑一个服务。

所以除了机器地址 IP，还要有“具体找哪个入口”的信息，这就是端口。

常见端口有：

| 端口 | 常见用途 |
| --- | --- |
| `22` | SSH 远程登录 |
| `80` | HTTP 网站访问 |
| `443` | HTTPS 网站访问 |
| `8000` | Django/Gunicorn 测试时常见端口 |

例如：

```text
1.2.3.4:22
```

表示：

```text
去 1.2.3.4 这台服务器的 22 号端口，也就是 SSH 入口。
```

再例如：

```text
1.2.3.4:8000
```

表示：

```text
去 1.2.3.4 这台服务器的 8000 号端口。
```

### 什么是“监听端口”

一个服务运行起来以后，必须“守在”某个端口上，外部请求才能找到它。

比如 Django 开发服务器运行在：

```text
0.0.0.0:8000
```

就表示它在服务器的 `8000` 端口等待请求。

如果一个服务根本没启动，或者启动了但没监听你访问的那个端口，浏览器当然访问不到。

### 什么是防火墙或安全组

就算服务已经在 `8000` 端口监听，也不代表外部一定能连进来。

因为服务器外面可能还有一道门禁：

- Linux 自己的防火墙
- 云服务器控制台里的安全组

它们的作用是决定：

```text
哪些端口允许外部访问，哪些端口禁止外部访问。
```

这也是为什么常见情况是：

- `ssh` 能连上
- 但网站打不开

因为：

```text
22 端口放行了，80 或 8000 端口没放行。
```

### 为什么正式网站通常只开放 80 和 443

正式部署时，最常见做法是：

- `80` 给 HTTP
- `443` 给 HTTPS
- `22` 给 SSH

而像 `8000` 这种端口：

- 开发测试可以临时开放
- 正式生产通常不对公网开放

原因是正式架构一般会变成：

```text
浏览器
  -> 80 或 443
    -> Nginx
      -> 服务器内部的 127.0.0.1:8000
```

也就是说：

```text
8000 往往是服务器内部自己用的，不直接暴露给外部。
```

### 一个非常常见的误区

很多人看到终端里服务已经启动，就以为浏览器一定能访问。

其实中间至少还有三层可能出问题：

1. 你访问的 IP 对不对
2. 你访问的端口对不对
3. 防火墙有没有放行

所以“服务启动了但浏览器访问不到”一点都不奇怪。

### 这一步最基础要确认什么

如果你现在有一台 Linux 服务器，这一步至少要确认：

1. 服务器公网 IP 是多少
2. 你能不能用这个 IP 成功 SSH 登录
3. 准备用哪个端口测试网站
4. 对应端口有没有放行

最简单的理解目标是：

```text
先搞清楚浏览器准备访问哪个地址、哪个端口。
```

### 小白版记忆方式

可以把访问网站理解成三连问：

1. 找没找到机器？
   也就是 IP 对不对

2. 找没找对门？
   也就是端口对不对

3. 门让不让进？
   也就是防火墙或安全组放没放行

### 这一章的检查问题

学完这一章，你应该能回答：

1. 公网 IP 是干什么的？
2. `80`、`443`、`22`、`8000` 这些端口大概各自做什么？
3. “服务已经启动，但浏览器打不开”时，至少可能是哪几类原因？
4. 为什么正式部署时通常不会把 `8000` 长期直接暴露给公网？

### 这一章的结论

这一章只要记住四句话：

1. 公网 IP 负责让别人找到你的服务器。
2. 端口负责告诉别人访问服务器上的哪个入口。
3. 服务启动了，不代表外部一定能访问，还要看防火墙和安全组。
4. 正式网站通常对外开放 `80` 和 `443`，而不是长期直接开放 `8000`。

## 4. 先用最简单方式在服务器跑通 Django 测试访问

### 这一章要解决什么问题

这一步只解决一个很朴素的问题：

```text
服务器上到底能不能把这个 Django/Wagtail 项目先跑起来，并从浏览器看到页面？
```

注意，这一步的目标不是正式上线。

这一步只是为了先验证三件事：

- 代码能不能在服务器上启动
- Python 环境和依赖有没有问题
- 浏览器能不能通过服务器 IP 先访问到页面

### 为什么这一步先不用 Gunicorn 和 Nginx

因为小白第一次部署时，最怕把很多层东西混在一起：

- Django 本身
- Gunicorn
- Nginx
- 域名
- HTTPS

如果一上来全配，出错时你不知道到底是哪一层坏了。

所以更稳的做法是：

```text
先只证明 Django 项目本身能在服务器上启动并被访问。
```

### 这一步的访问链路是什么

这一步用的是最简单的测试链路：

```text
浏览器
  -> 服务器IP:8000
    -> Django runserver
      -> Django / Wagtail
```

这和正式生产链路不同，但很适合最开始排基础问题。

### 为什么命令要写 `0.0.0.0:8000`

如果你在服务器上只写：

```bash
.venv/bin/python manage.py runserver
```

它通常只监听：

```text
127.0.0.1:8000
```

这意味着：

```text
只有服务器自己能访问这个服务。
```

如果你希望外部浏览器通过服务器 IP 访问，就要让 Django 开发服务器监听：

```text
0.0.0.0:8000
```

所以命令要写成：

```bash
.venv/bin/python manage.py runserver 0.0.0.0:8000
```

这里的 `0.0.0.0` 可以先理解成：

```text
不要只守在本机回环地址，而是接受这台服务器对外网卡上的请求。
```

小白阶段只需要记住区别：

| 写法 | 意义 |
| --- | --- |
| `127.0.0.1:8000` | 只有服务器自己能访问 |
| `0.0.0.0:8000` | 外部浏览器也有机会通过服务器 IP 访问 |

### `.venv` 是从哪来的

你会看到命令里写：

```bash
.venv/bin/python manage.py runserver 0.0.0.0:8000
```

这里的 `.venv` 是：

```text
项目自己的 Python 虚拟环境目录。
```

它不是 Django 自动帮你上传到服务器的，也不是 Git 仓库里应该提交的东西。

你这个项目当前 `.gitignore` 里已经忽略了：

```text
/.venv/
```

说明：

```text
.venv 不应该 push 到仓库。
```

正确理解是：

- 仓库里存代码和依赖清单
- 服务器上自己创建 `.venv`
- 再根据 `requirements.txt` 安装依赖

也就是说：

```text
.venv 是每台机器本地生成的运行环境，不是代码仓库内容。
```

### 这一步实际会做什么

如果服务器已经具备基本 Python 环境，并且代码已经在服务器上，那么这一步通常是：

1. SSH 登录服务器
2. 进入项目目录
3. 确认 `.venv` 存在
4. 用 `.venv/bin/python` 启动 Django 开发服务器
5. 浏览器访问 `http://服务器IP:8000`

### 如果 `.venv` 还不存在怎么办

那就说明服务器上还没准备好 Python 虚拟环境。

这时不是项目坏了，而是：

```text
服务器还没有创建它自己的运行环境。
```

后面会专门讲服务器环境和依赖安装。你现在先知道：

```text
.venv 不来自 Git，而来自这台机器上手工创建的虚拟环境。
```

### 这一步为什么只是“测试访问”

因为 `runserver` 是开发服务器，不是生产服务器。

所以这一步的目的只是：

```text
验证项目能跑
```

而不是：

```text
作为正式网站长期上线
```

这两件事一定要分开。

### 这一步成功后，你应该看到什么

如果成功，浏览器访问：

```text
http://服务器IP:8000
```

至少应该出现以下几种结果之一：

1. 正常打开网站页面
2. 页面打开了，但样式可能还不完整
3. Django 报错页，但说明请求已经真正到达 Django

这三种都比“完全打不开”更进一步，因为它们说明：

```text
浏览器已经成功连到服务器上的 Django 进程了。
```

### 如果还是打不开，最先怀疑什么

这一步如果失败，先不要想 Nginx。

优先检查：

1. 你是不是写了 `0.0.0.0:8000`
2. 服务器防火墙或安全组有没有放行 `8000`
3. Django 进程是不是其实没启动成功
4. 你访问的服务器 IP 对不对

### 这一章的检查问题

学完这一章，你应该能回答：

1. 为什么这一步先用 `runserver` 测试，而不是一上来就配 Nginx？
2. `0.0.0.0:8000` 和 `127.0.0.1:8000` 的区别是什么？
3. `.venv` 是什么？它该不该提交到 Git 仓库？
4. 如果浏览器访问 `http://服务器IP:8000` 失败，最先检查哪几类问题？

### 这一章的结论

这一章只要记住四句话：

1. 第一次部署时，先用最简单方式验证 Django 项目能不能在服务器跑起来。
2. `runserver 0.0.0.0:8000` 是为了让外部浏览器有机会访问服务器上的 Django。
3. `.venv` 是服务器本地创建的 Python 虚拟环境，不应该 push 到仓库。
4. 这一步只是临时测试链路，不是最终正式生产方案。

## 5. 理解为什么生产不用 `runserver`，而要 Gunicorn

### 这一章要解决什么问题

前一步我们故意用了最简单的方法：

```bash
.venv/bin/python manage.py runserver 0.0.0.0:8000
```

这一步要解决的问题是：

```text
既然这样已经能访问了，为什么正式部署时还要换成 Gunicorn？
```

### `runserver` 的定位是什么

`runserver` 是 Django 自带的开发服务器。

它最适合的场景是：

- 本地开发
- 本地调试
- 看改动效果
- 临时测试

它的设计目标是：

```text
方便开发者快速启动项目
```

不是：

```text
长期稳定地对公网提供正式网站服务
```

### 为什么 `runserver` 不适合正式生产

可以先不用背很细的原理，只记住几个现实层面的区别。

`runserver` 更像：

```text
临时拿来试一下的网站服务
```

而不是：

```text
专门为生产环境准备的正式 Web 服务进程
```

正式生产环境关心的是：

- 长时间稳定运行
- 更规范地处理并发请求
- 更适合和 Nginx 配合
- 更适合被 systemd 托管
- 更符合 Django/Wagtail 的生产部署习惯

### Gunicorn 是什么

Gunicorn 是一个 Python WSGI 服务器。

小白先不用死记 “WSGI” 这个词，只要先理解成：

```text
Gunicorn 是专门用来正式运行 Django 这类 Python Web 应用的服务程序。
```

它负责：

- 把 Django 项目正式启动起来
- 在指定端口等待请求
- 把请求交给 Django/Wagtail 代码处理
- 把结果返回出去

所以可以把它看成：

| 组件 | 小白理解 |
| --- | --- |
| `runserver` | 开发时自带的小测试服务器 |
| `Gunicorn` | 生产环境正式运行 Django 的服务器 |

### 为什么不是 Django 自己直接当正式服务器

这也是很多人一开始会疑惑的地方。

可以先这样理解：

```text
Django 是网站应用本身。
Gunicorn 是让 Django 以更正式方式跑起来的进程。
```

就像：

- Django 负责“网站逻辑和页面内容”
- Gunicorn 负责“把 Django 正式跑起来并接请求”

### 生产链路会怎么变化

临时测试时是：

```text
浏览器
  -> 服务器IP:8000
    -> Django runserver
```

正式运行时会先变成：

```text
浏览器
  -> 服务器某个端口
    -> Gunicorn
      -> Django / Wagtail
```

再下一步才会加上 Nginx，变成：

```text
浏览器
  -> Nginx
    -> Gunicorn
      -> Django / Wagtail
```

### 一个很重要的学习顺序

这里要特别分清：

1. `runserver`
   用来证明“项目能跑”

2. `Gunicorn`
   用来进入“正式运行 Django”的方式

3. `Nginx`
   用来作为公网入口和反向代理

也就是说：

```text
runserver 不是 Nginx 的替代品。
Gunicorn 也不是 Nginx 的替代品。
它们的分工不一样。
```

### 这一步的目标不是背概念，而是分清角色

你现在只需要把三者角色分清：

| 名称 | 角色 |
| --- | --- |
| Django/Wagtail | 网站应用本身 |
| `runserver` | 开发测试时临时启动 Django 的方式 |
| Gunicorn | 生产环境正式运行 Django 的方式 |

下一步我们才会真正开始用 Gunicorn 启动这个项目。

### 这一章的检查问题

学完这一章，你应该能回答：

1. `runserver` 更适合开发还是正式生产？
2. Gunicorn 的核心作用是什么？
3. 为什么“能用 `runserver` 打开页面”不等于“适合正式上线”？
4. Django、`runserver`、Gunicorn 三者的角色分别是什么？

### 这一章的结论

这一章只要记住四句话：

1. `runserver` 是开发服务器，适合本地调试和临时测试。
2. Gunicorn 是生产环境中更正式的 Django 运行方式。
3. Django 是网站应用本身，Gunicorn 是让它正式跑起来的服务进程。
4. 正式部署时，应该从 `runserver` 过渡到 Gunicorn。

## 6. 在服务器上用 Gunicorn 跑起 `lab-web`

### 这一章要解决什么问题

前一步已经知道：

```text
正式生产不应该继续用 runserver，而应该改用 Gunicorn。
```

这一章要解决的是：

```text
Gunicorn 到底怎么启动这个项目？命令里的每一部分各是什么意思？
```

### 这个项目里 Gunicorn 怎么启动

这个项目常见的启动写法是：

```bash
.venv/bin/gunicorn labsite.wsgi:application --bind 127.0.0.1:8000 --workers 3
```

这条命令第一次看会觉得很长，但其实可以拆开理解。

### 先解释 `.venv/bin/gunicorn`

这里的：

```text
.venv/bin/gunicorn
```

表示：

```text
使用这个项目虚拟环境里的 gunicorn 程序。
```

为什么不直接写 `gunicorn`？

因为一台服务器上可能有很多 Python 环境。写成：

```text
.venv/bin/gunicorn
```

意思更明确：

```text
就用这个项目自己的虚拟环境。
```

### `.venv` 是从哪来的，要不要 push 到仓库

这是你文档里明确标出来的问题，这里直接答。

`.venv` 是：

```text
项目所在机器上创建出来的 Python 虚拟环境目录。
```

它不是仓库里要提交的代码。

你这个项目当前 `.gitignore` 已经忽略了：

```text
/.venv/
```

所以结论是：

```text
.venv 不要 push 到仓库。
```

正确做法是：

1. 代码仓库里保存 `requirements.txt`
2. 服务器上自己创建 `.venv`
3. 在服务器上安装依赖

也就是说：

```text
仓库负责记录“需要哪些依赖”
.venv 负责在当前这台机器上真正安装这些依赖
```

### 再解释 `labsite.wsgi:application`

这部分表示：

```text
让 Gunicorn 去加载 Django 的 WSGI 应用入口。
```

可以先粗略理解成：

- `labsite.wsgi` 是 Python 模块路径
- `application` 是这个模块里暴露出来的 Django 应用对象

你这个项目里对应文件就是：

`labsite/wsgi.py`

Gunicorn 通过它知道：

```text
要启动的是哪个 Django 项目
```

### 再解释 `--bind 127.0.0.1:8000`

这部分是你文档里第二个明确标出来的问题。

`bind` 的意思可以先理解成：

```text
让 Gunicorn 守在哪个地址和端口上等待请求。
```

所以：

```text
--bind 127.0.0.1:8000
```

表示：

```text
Gunicorn 运行在本机回环地址的 8000 端口。
```

更直接一点说：

```text
Gunicorn 现在监听的是服务器自己的 127.0.0.1:8000
```

### 这和 `runserver 0.0.0.0:8000` 有什么区别

你问得很对，这两个看起来都在 `8000` 端口。

区别不在于“是不是 8000”，而在于：

```text
监听的是哪个地址
```

对比一下：

| 写法 | 谁能访问 |
| --- | --- |
| `runserver 0.0.0.0:8000` | 外部浏览器有机会直接访问 |
| `--bind 127.0.0.1:8000` | 只有服务器自己能访问 |

所以：

```text
是的，现在确实是 Gunicorn 运行在 8000 端口。
```

但要补完整一句：

```text
它运行在服务器自己的 127.0.0.1:8000 上，不是直接对公网开放。
```

### 为什么 Gunicorn 这里故意只绑到 `127.0.0.1`

因为正式架构通常是：

```text
浏览器
  -> Nginx
    -> 127.0.0.1:8000 上的 Gunicorn
```

也就是说：

- Nginx 对外接客
- Gunicorn 在服务器内部接 Nginx 转发过来的请求

这就是为什么 Gunicorn 通常不直接绑到：

```text
0.0.0.0:8000
```

因为它本来就不是打算直接对外开放的。

### 再解释 `--workers 3`

这个参数表示：

```text
启动 3 个工作进程处理请求。
```

小白阶段先不用深究“几个 workers 最佳”。

先知道它的作用是：

```text
不是只开一个单独进程傻等，而是开多个工作进程处理请求。
```

以后正式调优时，可以再根据服务器 CPU、内存、并发量调整。

### 这一步启动成功后说明什么

如果这条命令能成功跑起来，至少说明：

- Gunicorn 已安装在 `.venv` 里
- Django 项目能通过 `labsite.wsgi` 正常加载
- 项目基本能以生产方式启动

这时外部浏览器通常还不能直接打开页面，因为它绑的是：

```text
127.0.0.1:8000
```

但服务器自己可以访问它。

### 这一步和前一步的关系

前一步：

```text
runserver 0.0.0.0:8000
```

重点是“先让外部能粗暴测通”。

这一步：

```text
gunicorn --bind 127.0.0.1:8000
```

重点是“把 Django 切换到正式运行方式，为 Nginx 做准备”。

### 这一章的检查问题

学完这一章，你应该能回答：

1. `.venv` 是什么？要不要 push 到仓库？
2. `labsite.wsgi:application` 是干什么的？
3. `--bind 127.0.0.1:8000` 的意思是什么？
4. 为什么 Gunicorn 明明也用 `8000`，却和 `runserver 0.0.0.0:8000` 不一样？
5. `--workers 3` 大概是在干什么？

### 这一章的结论

这一章只要记住五句话：

1. `.venv` 是当前机器本地创建的 Python 虚拟环境，不属于 Git 仓库内容。
2. `labsite.wsgi:application` 告诉 Gunicorn 去加载这个 Django 项目的入口。
3. `--bind` 表示 Gunicorn 监听哪个地址和端口。
4. Gunicorn 这里虽然也用 `8000`，但因为绑在 `127.0.0.1`，它是给服务器内部用的。
5. 这一步的目的，是把项目切换到更接近正式生产的运行方式。

## 7. 理解为什么需要 Nginx 作为公网入口

### 这一章要解决什么问题

前一步已经让 Gunicorn 运行在：

```text
127.0.0.1:8000
```

这一章要解决的是：

```text
既然 Gunicorn 已经能跑 Django 了，为什么还要再加一个 Nginx？
```

### 先说结论

小白可以先记住一句最直白的话：

```text
Nginx 是对外接待浏览器的门口，Gunicorn 是服务器内部真正运行 Django 的人。
```

两者分工不同，不是重复造轮子。

### 如果没有 Nginx，会发生什么

如果你不加 Nginx，最粗暴的方式就是：

- 直接让 Gunicorn 对公网开放
- 或者继续让 `runserver` 对公网开放

技术上在某些场景下不是完全不行，但这不是标准、稳妥的正式部署方式。

因为正式网站除了“返回页面”之外，还要处理很多事情：

- 对外监听 80 和 443
- 处理域名
- 处理 HTTPS
- 反向代理到 Gunicorn
- 有时直接提供静态文件或媒体文件

这些事更适合交给 Nginx。

### Nginx 和 Gunicorn 的分工

你可以先这样理解：

| 组件 | 作用 |
| --- | --- |
| Nginx | 对公网开放，接收浏览器请求 |
| Gunicorn | 在服务器内部正式运行 Django |
| Django/Wagtail | 真正决定页面内容和业务逻辑 |

所以访问链路会变成：

```text
浏览器
  -> Nginx
    -> Gunicorn
      -> Django / Wagtail
```

### 什么叫“反向代理”

你后面会经常看到一个词：

```text
反向代理
```

先不要怕这个词。

可以先把它理解成：

```text
浏览器其实是在访问 Nginx，
Nginx 再把请求转交给 Gunicorn。
```

也就是说，浏览器表面上看到的是 Nginx，真正生成网页内容的是后面的 Django/Gunicorn。

### 为什么 Nginx 更适合站在最外层

因为对正式网站来说，公网入口通常需要更稳定地处理这些事情：

1. 接收外部请求
2. 根据域名决定把请求交给哪个站点
3. 处理 HTTP/HTTPS
4. 给 Gunicorn 转发请求头
5. 有时直接处理静态文件和上传文件

Nginx 在这类事情上很常见，也很适合作为最外层入口。

### 为什么 Gunicorn 绑到 `127.0.0.1:8000` 正好适合 Nginx

前一步我们已经把 Gunicorn 绑到了：

```text
127.0.0.1:8000
```

这正好说明一种正式架构思路：

- Gunicorn 不直接暴露给公网
- 只有服务器内部的 Nginx 去访问它

链路就变成：

```text
外部浏览器
  -> 服务器 80 / 443
    -> Nginx
      -> 服务器内部 127.0.0.1:8000
        -> Gunicorn
```

这比“直接把 Django/Gunicorn 裸露到公网”更规范。

### 这一章和下一章的关系

这一章只解决：

```text
为什么需要 Nginx
```

下一章才会解决：

```text
Nginx 配置怎么写，才能通过服务器 IP 访问网站
```

先把“为什么”搞清楚，再看配置文件，会更容易理解。

### 这一章的检查问题

学完这一章，你应该能回答：

1. 为什么 Gunicorn 跑起来以后，还不是最终完整的公网部署方案？
2. Nginx 和 Gunicorn 分别负责什么？
3. 什么叫反向代理？
4. 为什么正式架构里常见的是“浏览器 -> Nginx -> Gunicorn -> Django”？

### 这一章的结论

这一章只要记住四句话：

1. Nginx 是公网入口，Gunicorn 是服务器内部正式运行 Django 的进程。
2. 反向代理就是浏览器先访问 Nginx，再由 Nginx 把请求转给 Gunicorn。
3. 正式网站通常把域名、80/443、HTTPS 这些外层工作交给 Nginx。
4. Gunicorn 绑在 `127.0.0.1:8000`，正好适合让 Nginx 在服务器内部转发给它。

## 8. 配置 Nginx，让浏览器通过服务器 IP 访问网站

### 这一章要解决什么问题

前一步已经知道：

```text
Nginx 要站在最外层接收浏览器请求。
```

这一章要解决的是：

```text
Nginx 配置到底怎么写？每一行在干什么？
```

先说明，这一步先只讲：

```text
通过服务器 IP 访问
```

还不涉及正式域名和 HTTPS。

### 最简单的 Nginx 反向代理示例

一个最基础的配置可以长这样：

```nginx
server {
    listen 80;
    server_name 服务器IP;

    location / {
        proxy_pass http://127.0.0.1:8000;
        proxy_set_header Host $host;
        proxy_set_header X-Real-IP $remote_addr;
        proxy_set_header X-Forwarded-For $proxy_add_x_forwarded_for;
        proxy_set_header X-Forwarded-Proto $scheme;
    }
}
```

你文档里标的那个问题就是：

```text
这些每一行到底啥意思？
```

下面就一行一行拆。

### `server { ... }` 是什么

```nginx
server {
    ...
}
```

可以先理解成：

```text
定义一个站点的处理规则。
```

也就是：

```text
什么请求进来时，用这一套规则来处理。
```

以后一台服务器可以有多个 `server` 块，对应多个站点。

### `listen 80;` 是什么

```nginx
listen 80;
```

意思是：

```text
这个站点规则监听 80 端口。
```

也就是浏览器访问：

```text
http://服务器IP
```

时，请求会来到这里。

因为 HTTP 默认端口就是 `80`，所以浏览器通常不用手动写 `:80`。

### `server_name 服务器IP;` 是什么

```nginx
server_name 服务器IP;
```

意思是：

```text
当请求目标是这个 IP（或以后某个域名）时，用这个 server 块处理。
```

当前我们还没讲域名，所以先可以把它理解成：

```text
别人访问这个服务器 IP 时，Nginx 用这份配置来接待。
```

以后如果有正式域名，这里就会改成：

```nginx
server_name lab.example.com;
```

### `location / { ... }` 是什么

```nginx
location / {
    ...
}
```

意思是：

```text
当访问路径以 / 开头时，用里面这套规则处理。
```

因为网站大多数页面路径都以 `/` 开头，例如：

- `/`
- `/news/`
- `/admin/`

所以这个写法基本相当于：

```text
大部分请求都交给这里处理。
```

### `proxy_pass http://127.0.0.1:8000;` 是什么

```nginx
proxy_pass http://127.0.0.1:8000;
```

这行是最核心的。

意思是：

```text
Nginx 自己收到请求后，不自己生成页面，
而是把请求转发给服务器内部的 127.0.0.1:8000。
```

而 `127.0.0.1:8000` 上现在跑的是：

```text
Gunicorn
  -> Django / Wagtail
```

这就是“反向代理”真正落地的一行配置。

### `proxy_set_header Host $host;` 是什么

```nginx
proxy_set_header Host $host;
```

意思是：

```text
把浏览器原本访问时带的 Host 信息继续传给后面的 Gunicorn / Django。
```

小白阶段可以把 `Host` 理解成：

```text
浏览器说“我本来是来访问哪个 IP 或域名的”
```

把这个信息传进去很重要，因为 Django 后面要用它做：

- `ALLOWED_HOSTS` 检查
- 域名判断
- 站点识别

### `proxy_set_header X-Real-IP $remote_addr;` 是什么

```nginx
proxy_set_header X-Real-IP $remote_addr;
```

意思是：

```text
把真实访客的来源 IP 告诉后面的 Django。
```

否则后面的应用可能只看到“请求来自 Nginx”，却不知道真正访客是谁。

### `proxy_set_header X-Forwarded-For $proxy_add_x_forwarded_for;` 是什么

```nginx
proxy_set_header X-Forwarded-For $proxy_add_x_forwarded_for;
```

这也是传递访客来源信息的一种标准写法。

小白阶段不用细抠协议细节，只要知道：

```text
它能帮助后面的应用保留用户真实来源 IP 链路。
```

### `proxy_set_header X-Forwarded-Proto $scheme;` 是什么

```nginx
proxy_set_header X-Forwarded-Proto $scheme;
```

意思是：

```text
告诉后面的 Django，用户最外层访问时用的是 http 还是 https。
```

这行在以后 HTTPS 阶段尤其重要，因为 Django 需要知道：

```text
这个请求最外层到底是不是 HTTPS
```

你这个项目配置里有：

```text
SECURE_PROXY_SSL_HEADER = ('HTTP_X_FORWARDED_PROTO', 'https')
```

这就说明 Django 依赖这个头来理解代理后的 HTTPS 状态。

### 这一整段配置连起来是什么意思

把上面几行连起来翻译成人话，就是：

```text
Nginx 监听 80 端口；
当别人访问这个服务器 IP 时；
把大部分请求都转发给服务器内部 127.0.0.1:8000 的 Gunicorn；
并且把原始的 Host、IP、协议这些信息一并带过去。
```

### 这一步完成后，访问链路是什么

完成后链路会变成：

```text
浏览器
  -> http://服务器IP
    -> Nginx:80
      -> Gunicorn:127.0.0.1:8000
        -> Django / Wagtail
```

这时浏览器不需要写：

```text
:8000
```

因为浏览器访问的是标准 HTTP 入口 `80`，由 Nginx 负责往后转发。

### 这一步为什么比直接访问 `:8000` 更接近正式部署

因为它开始符合正式网站的常见结构：

- 浏览器只接触 Nginx
- Django 不直接站在最前面
- Gunicorn 退到服务器内部

这就是后面加域名、加 HTTPS 的基础。

### 这一章的检查问题

学完这一章，你应该能回答：

1. `server {}` 大概是在定义什么？
2. `listen 80;` 是什么意思？
3. `server_name` 在这里起什么作用？
4. `location / {}` 大概管哪些请求？
5. `proxy_pass http://127.0.0.1:8000;` 为什么是整段配置里最核心的一行？
6. 几个 `proxy_set_header` 大概是在干什么？

### 这一章的结论

这一章只要记住五句话：

1. Nginx 的 `server` 块是在定义一个站点的接待规则。
2. `listen 80` 表示这个站点接收标准 HTTP 请求。
3. `proxy_pass` 的作用是把请求转发给后面的 Gunicorn。
4. `proxy_set_header` 的作用是把原始请求的重要信息带给 Django。
5. 做完这一步后，浏览器访问的是服务器 IP 的 80 端口，而不是直接访问 `:8000`。

## 9. 理解静态文件 `static` 和上传文件 `media` 的区别

### 这一章要解决什么问题

这一步要解决一个特别容易混的点：

```text
static、static_src、static_compiled、media 到底分别是什么？
```

你文档里标出来的问题也正好在这里：

```text
static 不是有编译吗？这是编译前的还是编译后的？要在开发环境编译还是在生产环境重新编译？
```

这一章就专门把这个问题拆清楚。

### 先说最核心结论

这个项目里至少有四种容易混在一起的东西：

| 名称 | 作用 |
| --- | --- |
| `static_src/` | 前端源码，给开发者改的 |
| `static_compiled/` | 前端编译结果 |
| `static/` | Django `collectstatic` 统一收集后的静态文件目录 |
| `media/` | 后台上传文件目录 |

如果这四个不区分，后面你会非常容易搞乱。

### `static_src/` 是什么

`static_src/` 是：

```text
前端源码目录。
```

也就是你平时开发时会改的这些东西：

- Sass
- JavaScript
- Tailwind 相关源码

可以理解成：

```text
这是开发者写的原材料，还不是浏览器最终直接吃的成品。
```

### `static_compiled/` 是什么

`static_compiled/` 是：

```text
前端源码经过构建工具编译出来的结果。
```

在你这个项目里，大致链路是：

```text
static_src/
  -> npm run build 或 npm run build:prod
    -> static_compiled/
```

也就是说：

```text
static_compiled 是“编译后的静态资源”
```

比如编译后的：

- CSS
- JS

会在这里。

### `static/` 是什么

`static/` 不是你手工开发时主要去改的地方。

它是 Django 在生产环境里，通过：

```bash
manage.py collectstatic
```

统一收集出来的静态文件目录。

可以把它理解成：

```text
Django 最终准备对外提供静态文件时的汇总目录。
```

来源不只包括你自己的 `static_compiled/`，还包括：

- Django 自带后台静态文件
- Wagtail 后台静态文件
- 第三方 app 的静态文件

所以：

```text
static_compiled 是你项目前端构建的产物
static 是 Django 最终统一收集后的对外静态目录
```

### `media/` 又是什么

`media/` 完全不是一回事。

它是：

```text
后台用户上传文件存放的目录。
```

比如：

- 上传的新闻图片
- 教师照片
- 附件文件

这些通常不是开发者写在代码里的，也不是前端编译出来的。

它们是网站运行过程中，由后台操作产生的内容。

### 用一个表彻底区分

| 目录 | 谁产生的 | 主要存什么 | 要不要进 Git |
| --- | --- | --- | --- |
| `static_src/` | 开发者 | 前端源码 | 要 |
| `static_compiled/` | 构建命令 | 编译后的 CSS/JS | 视项目习惯而定 |
| `static/` | `collectstatic` | Django 最终收集的静态文件 | 通常不要 |
| `media/` | 后台用户上传 | 图片、附件等媒体文件 | 通常不要 |

### 你问的“static 是编译前还是编译后”该怎么回答

严格说要分情况：

1. `static_src/`
   是编译前

2. `static_compiled/`
   是前端编译后

3. `static/`
   是 Django 再收集后的最终静态目录

所以不能笼统说 “static” 一定是编译前还是编译后，要看具体指的是哪个目录。

### 在开发环境编译，还是在生产环境重新编译

对你这个项目，更稳的理解方式是：

```text
开发环境可以编译，用来本地看效果。
生产环境也应该重新构建一次，用来生成服务器自己的正式静态资源。
```

为什么？

因为生产服务器最终真正运行的网站，依赖的是：

- 服务器当前代码版本
- 服务器当前安装的前端依赖
- 服务器当前执行出的构建产物

所以即使你本地编译过，生产部署时通常仍然会做：

```bash
npm run build:prod
```

这样更稳。

### 为什么你平时本地只记得 `npm run build`

因为本地开发时，你主要是在解决：

```text
浏览器要看到新的 CSS/JS 效果
```

所以你常做的是前端构建：

```bash
npm run build
```

但本地开发不一定每次都需要：

```bash
manage.py collectstatic
```

因为本地 `runserver` 和开发调试场景，很多时候不靠“最终生产静态汇总目录”来运作。

而生产环境不同，它更依赖完整、规范的静态文件流程。

### 这一章先记住的核心链路

你这个项目的静态资源链路，可以先记成：

```text
static_src/
  -> 前端构建
    -> static_compiled/
      -> collectstatic
        -> static/
```

而上传文件是另一条线：

```text
后台上传
  -> media/
```

这两条线完全不要混在一起。

### 这一章的检查问题

学完这一章，你应该能回答：

1. `static_src/`、`static_compiled/`、`static/`、`media/` 分别是什么？
2. 哪些是开发者代码产物，哪些是后台上传内容？
3. 为什么 `media/` 不能和 `static` 混为一谈？
4. 为什么生产环境通常还要重新执行前端构建？

### 这一章的结论

这一章只要记住五句话：

1. `static_src/` 是前端源码，`static_compiled/` 是前端编译结果。
2. `static/` 是 Django `collectstatic` 后的最终静态汇总目录。
3. `media/` 是后台上传文件目录，和静态资源不是一回事。
4. 本地开发时常做前端构建，生产部署时通常也要在服务器重新构建一次。
5. 记住两条链路：`static_src -> static_compiled -> static`，以及 `后台上传 -> media`。

## 10. 在服务器上构建前端并执行 `collectstatic`

### 这一章要解决什么问题

这一步专门解决一个很常见的困惑：

```text
我本地平时只记得跑 npm run build，
为什么到了生产环境还要跑 manage.py collectstatic？
```

答案是：

```text
这两个命令根本不是在做同一件事。
```

### 先说最核心区别

1. 前端构建命令，例如：

   ```bash
   npm run build
   npm run build:prod
   ```

   作用是：

   ```text
   把前端源码编译成 CSS / JS 产物。
   ```

2. Django 静态收集命令：

   ```bash
   DJANGO_SETTINGS_MODULE=labsite.settings.production .venv/bin/python manage.py collectstatic --noinput
   ```

   作用是：

   ```text
   把 Django/Wagtail/项目自身需要对外提供的静态文件统一收集到 STATIC_ROOT。
   ```

所以：

```text
build 是“前端编译”
collectstatic 是“Django 统一收集静态文件”
```

### 为什么本地开发时你常常只记得 `npm run build`

因为本地开发时，你最直观的需求通常是：

```text
我改了 Sass / JS，要赶紧看到页面效果。
```

这时前端构建就够你感知变化了。

但正式生产环境不一样。

生产环境不只是要“编译出前端文件”，还要让 Django 进入一个更规范的静态资源提供流程，所以才会多出：

```bash
collectstatic
```

### `collectstatic` 到底在干什么

可以把它理解成：

```text
把各个地方分散的静态文件，收拢到一个最终目录里。
```

来源包括：

- 你自己的 `static_compiled/`
- Django admin 自带静态资源
- Wagtail 后台静态资源
- 第三方 app 静态资源

这些东西平时分散在不同位置，但生产环境需要一个最终汇总目录。

在你这个项目里，这个最终目录就是：

```text
static/
```

因为配置里有：

```text
STATIC_ROOT = static/
```

### 为什么不能只做前端构建，不做 `collectstatic`

因为浏览器访问生产网站时，不只是要用你自己写的 CSS/JS。

它还可能需要：

- Django 自带后台静态文件
- Wagtail 后台静态文件
- 第三方组件静态文件

如果你只做：

```bash
npm run build:prod
```

却不做：

```bash
collectstatic
```

就可能出现：

- 前台部分资源正常
- 后台样式错乱
- 某些静态文件 404
- 更新后的静态文件没有被正确收集

### 为什么命令前面要写 `DJANGO_SETTINGS_MODULE=...`

你看到的命令是：

```bash
DJANGO_SETTINGS_MODULE=labsite.settings.production .venv/bin/python manage.py collectstatic --noinput
```

这里的：

```text
DJANGO_SETTINGS_MODULE=labsite.settings.production
```

表示：

```text
这次命令明确使用生产配置来执行。
```

因为 `collectstatic` 依赖 Django 的配置项，比如：

- `STATIC_ROOT`
- `STATIC_URL`
- `STORAGES`
- 其他静态文件相关设置

所以要明确告诉 Django：

```text
现在按 production 配置来收集静态文件。
```

### `--noinput` 是什么意思

`--noinput` 表示：

```text
执行过程中不要停下来问交互问题，直接按默认方式继续。
```

这在部署脚本里很常见，因为正式发布时不希望命令跑一半停住等人手动输入。

### 这一步完整链路是什么

你这个项目在服务器上的静态资源处理链路，可以理解成：

```text
static_src/
  -> npm run build:prod
    -> static_compiled/
      -> manage.py collectstatic
        -> static/
```

这里每一层的角色不同：

| 环节 | 作用 |
| --- | --- |
| `static_src/` | 前端源码 |
| `npm run build:prod` | 编译前端 |
| `static_compiled/` | 你项目自己的编译产物 |
| `collectstatic` | 统一收集所有静态资源 |
| `static/` | 最终生产静态目录 |

### 为什么生产环境一般要重新跑一遍

因为生产服务器最终实际提供给用户的内容，应该由生产服务器自己当前的代码和依赖生成。

所以更稳妥的部署习惯是：

```bash
npm install
npm run build:prod
DJANGO_SETTINGS_MODULE=labsite.settings.production .venv/bin/python manage.py collectstatic --noinput
```

而不是完全依赖你本地电脑提前生成的结果。

### 如果漏掉其中一步，常见表现是什么

1. 漏掉前端构建

   常见表现：

   - 新改的样式没生效
   - 新改的 JS 没生效

2. 漏掉 `collectstatic`

   常见表现：

   - 某些静态资源 404
   - Wagtail 后台样式异常
   - 生产环境仍在使用旧静态文件

### 这一章的检查问题

学完这一章，你应该能回答：

1. `npm run build:prod` 和 `collectstatic` 分别做什么？
2. 为什么本地开发时常觉得只跑 `build` 就够了？
3. 为什么生产环境通常还要执行 `collectstatic`？
4. `DJANGO_SETTINGS_MODULE=labsite.settings.production` 在这个命令前面起什么作用？
5. `--noinput` 大概是什么意思？

### 这一章的结论

这一章只要记住五句话：

1. `npm run build:prod` 负责把前端源码编译成项目自己的 CSS/JS 产物。
2. `collectstatic` 负责把 Django/Wagtail/第三方 app/项目自身的静态文件统一收集起来。
3. 本地开发时常只感觉到前端构建，生产环境则更依赖完整静态收集流程。
4. 生产环境执行 `collectstatic` 时要明确使用 `production` 配置。
5. 对你这个项目，标准顺序是先构建前端，再执行 `collectstatic`。

## 11. 确认通过 IP 访问时页面样式、后台、图片都正常

### 这一章要解决什么问题

前面已经有了这些基础：

- 服务器能运行项目
- Gunicorn 能启动
- Nginx 能做代理
- 前端构建和 `collectstatic` 已执行

这一章要解决的是：

```text
现在通过服务器 IP 访问时，到底怎样才算“真的跑通了”？
```

不是看到首页打开就算完事，还要检查：

- 页面内容
- 页面样式
- 后台登录
- 上传图片
- 图片显示

### 为什么不能只看“首页能打开”

因为“首页能打开”只说明链路的最表层通了。

但它不代表下面这些都正常：

- 静态文件是否正常
- Wagtail 后台是否正常
- 上传图片路径是否正常
- `media/` 是否能被访问

所以这一步本质上是在做：

```text
部署后的第一轮功能验证
```

### 这一步应该怎么验证

最基本的检查顺序可以是：

1. 打开首页
2. 点开几个栏目页
3. 打开后台 `/admin/`
4. 登录后台
5. 看后台样式是否正常
6. 找一张已有图片，看前台能不能显示
7. 在后台尝试上传一张测试图片
8. 上传后看前台或后台缩略图是否能显示

### 页面样式正常，说明了什么

如果页面样式正常，通常说明这些环节基本没大问题：

- 前端构建完成了
- `collectstatic` 做了
- 静态文件路径没完全坏
- Django/Wagtail 能正常引用 CSS/JS

如果页面能打开但样式乱掉，通常优先怀疑：

- 忘了 `npm run build:prod`
- 忘了 `collectstatic`
- Nginx / Django 没正确提供静态文件

### 后台 `/admin/` 正常，说明了什么

Wagtail 后台是一个很好的检测点。

如果后台能正常打开，而且样式也正常，说明：

- Django 本身运行正常
- Wagtail 后台静态文件基本正常
- 账号登录流程基本正常

如果后台页面能打开但样式乱掉，说明：

```text
应用本身可能还活着，但静态文件链路不完整。
```

### 图片能显示，说明了什么

图片显示正常，通常说明：

- 数据库里保存的图片记录没问题
- `media/` 目录里有对应文件
- Nginx 或 Django 能正确提供这些媒体文件

反过来，如果页面文字都正常，但图片全裂了，通常优先怀疑：

- `media/` 没迁过来
- 路径不对
- Nginx 没正确处理 `/media/`

### 你问的“怎么确认 Nginx 能访问 `/media/`”

这是你文档里明确标出来的问题，这里直接讲清楚。

所谓：

```text
Nginx 能访问 /media/
```

不是说你在终端里问 Nginx 一句“你能不能访问”。

真正的意思是：

```text
浏览器请求 /media/ 下的文件时，
Nginx 能不能把服务器磁盘里的对应文件正确返回给浏览器。
```

最直观的确认方法就是：

1. 后台里找一张已经上传的图片
2. 在前台或后台缩略图中查看它是否正常显示

如果图片地址类似：

```text
http://服务器IP/media/images/xxx.jpg
```

你把这个地址直接打开：

- 能显示图片，说明 `/media/` 基本通了
- 打不开或 404，说明 `/media/` 这条链路有问题

所以判断方式不是抽象地猜，而是：

```text
直接请求一个真实的 media 文件 URL 看能不能返回
```

### `/media/` 失败时常见意味着什么

如果 `/media/` 地址打不开，常见原因有：

1. 服务器上压根没有对应文件
2. `media/` 路径配置不对
3. Nginx 没配置 `location /media/`
4. Nginx 的 `alias` 或路径写错了
5. 权限问题导致 Nginx 读不到文件

### 这一步为什么在“迁移数据库之前”也值得做

即使你还没迁正式数据，这一步也值得做。

因为你是在确认整条访问链路：

- 浏览器到 Nginx
- Nginx 到 Gunicorn
- Gunicorn 到 Django
- 静态文件
- 媒体文件

如果这些基础链路不先跑顺，后面把数据库迁上去，只会让问题更多。

### 这一步最小通过标准

小白阶段，一个比较实用的“通过标准”可以是：

1. 用服务器 IP 打开首页
2. 页面样式正常
3. `/admin/` 能打开并登录
4. 后台样式正常
5. 至少一张图片能正常显示

只要这五件事过了，说明你前半段部署链路已经基本成型。

### 这一章的检查问题

学完这一章，你应该能回答：

1. 为什么不能只因为首页能打开就认为部署完成？
2. 页面样式正常，大概说明哪条链路没出大问题？
3. 后台样式异常通常优先怀疑什么？
4. 怎样最直接地确认 `/media/` 这条链路是否正常？
5. 页面文字正常但图片裂了，优先应该想到哪几类原因？

### 这一章的结论

这一章只要记住五句话：

1. 部署验证不能只看首页是否能打开，还要看样式、后台和图片。
2. 页面样式正常通常说明静态文件链路基本没大问题。
3. 后台能登录且样式正常，是 Django/Wagtail 部署健康的重要信号。
4. 确认 `/media/` 是否正常，最直接的方法就是打开一个真实图片 URL 看能不能返回。
5. 只有页面、静态资源、后台、图片都基本正常，才能说明 IP 访问阶段真正跑通。

## 12. 理解当前 SQLite 数据库里存了什么

### 这一章要解决什么问题

接下来要讲数据迁移，但在迁移之前，先要回答一个基础问题：

```text
你现在这个 db.sqlite3 里到底装了什么？
```

如果不知道数据库里装了什么，就很容易误以为：

- 只拷代码就够了
- 或者只拷数据库就够了

其实都不对。

### `db.sqlite3` 是什么

你当前开发环境主要用的是：

```text
db.sqlite3
```

它是 SQLite 数据库文件。

可以先把它理解成：

```text
当前这个网站的大账本。
```

Wagtail 和 Django 很多核心内容都存这里。

### 这里面通常存哪些东西

对你这个项目来说，`db.sqlite3` 里通常会包含这些内容：

- 页面树结构
- 首页内容
- 新闻内容
- 师资信息
- 科研成果信息
- 联系方式
- 导航和站点设置
- Wagtail 后台用户
- Django 用户、权限、组
- 其他模型数据

也就是说，很多你在后台里“填进去的文字内容”，都在数据库里。

### 为什么说它是“网站内容的大脑”

因为模板只是“长什么样”。

但真正的页面内容，比如：

- 某条新闻标题
- 某个教师简介
- 某个科研成果描述
- 首页介绍文字

这些实际内容很多不写死在模板里，而是从数据库读出来的。

所以如果你只把代码传到服务器，却没有数据库，常见结果就是：

- 页面结构可能还在
- 但具体内容没有了
- 后台账号也没有了

### 图片信息和图片文件为什么是两回事

这里有一个很关键的点：

```text
数据库里有图片“记录”
不等于数据库里有图片“文件本体”
```

比如后台上传一张图片后，数据库里通常会存：

- 这张图片的记录
- 文件路径
- 标题、宽高等元信息

但真正的图片文件本体，通常是在：

```text
media/
```

所以这两部分必须一起理解：

| 东西 | 存在哪里 |
| --- | --- |
| 图片记录、路径、元信息 | 数据库 |
| 图片文件本体 | `media/` |

### 为什么“只有数据库”也不够

如果你只把 `db.sqlite3` 搬到服务器，但没有把 `media/` 一起搬过去，就会发生这种情况：

- 页面里还知道“这里本来有一张图”
- 但服务器磁盘上没有真正的图片文件
- 所以前台图片会裂掉，或者返回 404

这就是为什么部署迁移时不能只盯着数据库。

### 为什么“只有 media/”也不够

反过来也一样。

如果你只有 `media/` 目录，但没有数据库：

- 服务器上有一堆图片文件
- 但 Wagtail 不知道这些图片属于谁、挂在哪个页面上

所以仅有文件也不够。

### 代码、数据库、media 三者分别负责什么

可以先用一个最简单的表来记：

| 部分 | 负责什么 |
| --- | --- |
| 代码 | 页面结构、逻辑、模板、样式 |
| 数据库 | 页面内容、后台数据、记录关系 |
| `media/` | 上传文件本体 |

只有三者配合起来，网站才完整。

### 这一章和下一章的关系

这一章只解决：

```text
数据库里到底有什么
```

下一章才开始讲：

```text
怎么把本地 SQLite 和 media 迁到服务器
```

先分清“要迁移什么”，再讲“怎么迁”，会更容易理解。

### 这一章的检查问题

学完这一章，你应该能回答：

1. `db.sqlite3` 在当前项目里主要扮演什么角色？
2. 页面内容、后台账号、权限这些通常在数据库里还是在模板里？
3. 图片记录和图片文件本体为什么是两回事？
4. 为什么只迁数据库不够？
5. 为什么只迁 `media/` 也不够？

### 这一章的结论

这一章只要记住五句话：

1. `db.sqlite3` 保存的是网站的很多内容数据和后台数据，不只是“技术配置”。
2. 模板决定页面怎么长，数据库决定页面具体显示什么内容。
3. 图片记录通常在数据库里，图片文件本体通常在 `media/` 里。
4. 迁移网站时不能只迁代码，也不能只迁数据库。
5. 对当前项目来说，完整内容迁移至少要同时考虑代码、数据库和 `media/`。

## 13. 把本地 SQLite 和 `media/` 迁移到服务器作为过渡方案

### 这一章要解决什么问题

现在已经知道：

- 数据库里有很多内容
- `media/` 里有上传文件

这一章要解决的是：

```text
如果我现在就想先把现有网站搬到服务器，最简单的方法是什么？
```

答案就是一个很朴素的过渡方案：

```text
把本地的 db.sqlite3 和 media/ 一起迁到服务器。
```

### 为什么说这是“过渡方案”

因为它的优点是：

- 好理解
- 操作直接
- 不用先学 PostgreSQL
- 很适合第一次把站点先跑起来

但它不是长期最理想的正式方案，因为：

- SQLite 更适合开发和小规模过渡
- 以后正式长期运行通常还是更推荐 PostgreSQL

所以这一步的定位是：

```text
先把现有内容整体搬到服务器上，让站点先可用。
```

### 这一步到底迁什么

最小迁移集合通常是：

1. 代码
2. `db.sqlite3`
3. `media/`

也就是：

```text
代码决定站点结构和逻辑
db.sqlite3 决定内容数据
media/ 决定上传文件本体
```

### 这一步的核心思路

可以把它理解成：

```text
先让服务器暂时继续用 SQLite，
把你本地已经有的网站内容原样搬过去。
```

对于第一次上线来说，这往往是最容易理解的一步。

### 迁移之后服务器上会是什么状态

迁完以后，服务器上的项目会大致变成：

- 代码在服务器目录里
- 服务器项目目录里也有一个 `db.sqlite3`
- 服务器项目目录里也有一个 `media/`

然后 Django/Wagtail 在服务器上继续使用它们。

### 为什么一定要“数据库和 media 一起迁”

因为它们是一组配套数据。

如果只迁数据库：

- 页面里知道有图片
- 但图片文件不存在

如果只迁 `media/`：

- 文件存在
- 但数据库里没有对应记录

所以最朴素的正确心法是：

```text
把网站内容搬家时，数据库和 media 要一起搬。
```

### 这一步最容易踩的坑：覆盖问题

这一步最需要小心的是：

```text
服务器上原来有没有新的内容
```

如果服务器上已经有人在后台新建了页面、上传了图片、改了内容，而你又直接把本地的：

- `db.sqlite3`
- `media/`

整个覆盖过去，那服务器上的新内容就可能丢失。

所以这个方案最适合：

- 第一次把站点从本地搬到服务器
- 或者服务器上还没有新的生产数据

### 这一步迁移成功后应该看到什么

如果迁移成功，通常表现为：

- 前台页面内容和你本地一致
- 后台账号还能登录
- 原来已有的图片还能显示
- 新闻、师资、科研成果等内容都还在

如果迁完后页面结构在、内容不对，通常说明数据库没有同步好。

如果文字都在、图片全裂，通常说明 `media/` 没同步好。

### 这一步和前面验证步骤怎么配合

最稳妥的顺序通常是：

1. 先证明服务器代码链路能跑通
2. 再把 `db.sqlite3` 和 `media/` 搬过去
3. 再重新检查内容和图片

这样你能分清楚：

- 是“程序跑不起来”
- 还是“内容没搬对”

而不是所有问题混在一起。

### 这一步的实质

这一步本质上不是在做什么很神秘的“数据库迁移技术”。

它更像：

```text
把现有网站的数据文件和上传文件目录，一起搬到服务器继续使用。
```

所以它是小白最容易上手的过渡路径。

### 这一章的检查问题

学完这一章，你应该能回答：

1. 为什么把本地 SQLite 和 `media/` 搬到服务器可以作为过渡方案？
2. 这一步最小要迁哪几类东西？
3. 为什么数据库和 `media/` 要一起迁？
4. 这一步最大的风险是什么？
5. 如果迁完后文字正常但图片全裂，优先怀疑哪部分？

### 这一章的结论

这一章只要记住五句话：

1. 第一次搬站点时，直接迁本地 `db.sqlite3` 和 `media/` 是最容易理解的过渡方案。
2. 这个方案适合先把现有内容整体搬到服务器，不代表它是长期最优正式方案。
3. 完整内容迁移至少要同时考虑代码、数据库和 `media/`。
4. 这一步最大风险是覆盖掉服务器上已经产生的新内容。
5. 迁完后要重新验证前台内容、后台登录和图片显示是否与本地一致。

## 14. 理解为什么正式环境建议换 PostgreSQL

### 这一章要解决什么问题

前一步已经知道：

```text
先把 SQLite 和 media 搬到服务器，可以作为一个容易理解的过渡方案。
```

这一章要解决的是：

```text
既然 SQLite 也能用，为什么大家正式环境还经常推荐 PostgreSQL？
```

### 先说最核心结论

可以先用一句最简单的话记住：

```text
SQLite 更像“单文件数据库，适合开发和过渡”；
PostgreSQL 更像“正式数据库服务，适合长期生产运行”。
```

### SQLite 的特点是什么

SQLite 的一个最大特点是：

```text
整个数据库就是一个文件。
```

对你这个项目来说，就是：

```text
db.sqlite3
```

它的优点非常明显：

- 简单
- 轻便
- 不需要单独装数据库服务
- 本地开发特别省事

这也是为什么 Django/Wagtail 项目本地开发经常直接用 SQLite。

### PostgreSQL 的特点是什么

PostgreSQL 不是一个单独的 `.sqlite3` 文件。

它是：

```text
一个独立运行的数据库服务。
```

你的 Django 项目不是直接打开一个本地文件，而是通过数据库连接去访问它。

可以先理解成：

```text
SQLite 像“一个本地数据文件”
PostgreSQL 像“专门负责管数据的正式服务”
```

### 为什么正式环境更偏向 PostgreSQL

因为正式网站关心的不只是“能不能存数据”，还关心：

- 长期稳定运行
- 更规范地处理并发访问
- 更适合多人后台编辑
- 更规范的备份恢复方式
- 更清晰的数据库用户和权限管理

这些方面，PostgreSQL 通常更适合正式生产环境。

### 为什么对 Wagtail 网站尤其常见

Wagtail 站点往往不只是“一个静态展示页”。

它通常会有：

- 后台编辑
- 页面发布
- 图片上传
- 多种内容模型
- 后台用户和权限

这些东西长期跑起来后，数据库的重要性会越来越高。

所以：

```text
开发阶段用 SQLite 很自然；
正式长期运行时用 PostgreSQL 更常见、更稳妥。
```

### 这并不代表 SQLite 完全不能上线

这点也要讲清楚。

SQLite 不是“绝对不能上线”，而是：

```text
不太适合作为长期正式生产方案。
```

所以对于小白第一次部署来说：

```text
先用 SQLite 过渡上线
```

是能理解、也经常可行的。

只是如果你准备长期维护这个站点，或者希望部署更规范，后面还是建议切到 PostgreSQL。

### 所以这一步真正要学会的不是“SQLite 错了”

而是学会区分两个阶段：

1. 过渡阶段

   ```text
   先用 SQLite，把网站先跑起来。
   ```

2. 正式阶段

   ```text
   再换 PostgreSQL，让数据库部分更接近标准生产形态。
   ```

这就像：

```text
先学会走，再学会跑。
```

### 这一章和下一章的关系

这一章只解决：

```text
为什么建议换 PostgreSQL
```

下一章才开始讲：

```text
如果真的要换，SQLite 数据怎么迁过去
```

### 这一章的检查问题

学完这一章，你应该能回答：

1. SQLite 和 PostgreSQL 在定位上最大的区别是什么？
2. 为什么 SQLite 很适合本地开发？
3. 为什么正式长期运行的网站更常推荐 PostgreSQL？
4. “推荐 PostgreSQL” 是否等于 “SQLite 完全不能上线”？
5. 为什么说对小白来说，先 SQLite 过渡、后 PostgreSQL，是很合理的学习顺序？

### 这一章的结论

这一章只要记住五句话：

1. SQLite 是单文件数据库，简单轻便，适合开发和过渡部署。
2. PostgreSQL 是独立数据库服务，更适合长期正式生产环境。
3. 正式网站更关心稳定性、并发、备份和权限管理，这些方面 PostgreSQL 更常见。
4. SQLite 不是完全不能上线，但通常不是长期正式方案的首选。
5. 对当前项目，先 SQLite 过渡、后 PostgreSQL 迁移，是一条很适合小白的路线。

## 15. 把 SQLite 数据迁移到 PostgreSQL

### 这一章要解决什么问题

前一步讲了：

```text
正式环境建议使用 PostgreSQL。
```

这一章要解决的是：

```text
如果我现在已有 SQLite 数据，怎么理解把它迁到 PostgreSQL 这件事？
```

这里有两个你标出来的重点问题：

- `DATABASE_URL` 到底是什么，怎么设置？
- `migrate` 是不是直接就能把 SQLite 数据迁到 PostgreSQL？

### 先说最重要的结论

把 SQLite 换成 PostgreSQL，不是复制一个文件。

它至少包含四件不同的事：

1. 准备 PostgreSQL 数据库
2. 让 Django 知道要连接这个 PostgreSQL
3. 在 PostgreSQL 里创建表结构
4. 把 SQLite 里的已有内容数据导入 PostgreSQL

这四件事不能混成一件。

### `DATABASE_URL` 是什么

`DATABASE_URL` 是一个环境变量。

它的作用是：

```text
告诉 Django：生产环境应该连接哪个数据库。
```

你这个项目的配置里使用了 `dj_database_url`，所以可以通过 `DATABASE_URL` 来配置数据库。

一个 PostgreSQL 的 `DATABASE_URL` 通常长这样：

```text
postgres://用户名:密码@数据库主机:端口/数据库名
```

例如：

```text
postgres://labweb_user:your_password@127.0.0.1:5432/labweb
```

拆开看：

| 部分 | 意思 |
| --- | --- |
| `postgres://` | 使用 PostgreSQL |
| `labweb_user` | 数据库用户名 |
| `your_password` | 数据库密码 |
| `127.0.0.1` | 数据库运行在哪台机器 |
| `5432` | PostgreSQL 默认端口 |
| `labweb` | 数据库名 |

### 为什么要用环境变量，而不是写死在代码里

因为数据库密码、生产服务器地址这些信息不应该写进 Git 仓库。

所以更合理的做法是：

```text
代码里写“从环境变量读取数据库配置”
服务器环境里设置真正的 DATABASE_URL
```

这样：

- 代码可以安全提交
- 生产密码不会进仓库
- 不同环境可以用不同数据库

### `migrate` 到底做什么

这是你标出来的另一个关键问题。

命令：

```bash
DJANGO_SETTINGS_MODULE=labsite.settings.production .venv/bin/python manage.py migrate
```

不是把 SQLite 数据自动搬到 PostgreSQL。

它做的是：

```text
根据 Django migrations，在当前连接的数据库里创建或更新表结构。
```

也就是说：

```text
migrate 主要管“表结构”
不是自动搬“已有内容数据”
```

### 用一个例子理解 `migrate`

假设你的模型里有：

```text
新闻标题
新闻正文
发布日期
```

`migrate` 会负责在数据库里创建类似这样的表结构：

```text
这里有一张新闻表
这张表有标题字段
这张表有正文字段
这张表有发布日期字段
```

但它不会自动知道：

```text
你原来 SQLite 里已经有 100 条新闻，要帮你全部搬过去。
```

所以：

```text
migrate 是建房子结构
导入数据才是把家具搬进去
```

### 那已有 SQLite 数据怎么过去

常见思路是：

```text
从 SQLite 导出数据
  -> 切换到 PostgreSQL 配置
    -> migrate 建好表结构
      -> 把数据导入 PostgreSQL
```

Django 常见命令是：

```bash
python manage.py dumpdata
python manage.py loaddata
```

小白阶段先不用急着背完整参数，先理解它们的角色：

| 命令 | 作用 |
| --- | --- |
| `dumpdata` | 从当前数据库导出数据 |
| `loaddata` | 把导出的数据导入当前数据库 |
| `migrate` | 创建或更新数据库表结构 |

### 为什么 Wagtail 数据迁移要谨慎

Wagtail 项目数据关系比较多，比如：

- 页面树
- revisions
- images
- documents
- users
- permissions
- content types

所以把 SQLite 数据导入 PostgreSQL 前，一定要先备份：

- `db.sqlite3`
- `media/`

不要在没有备份的情况下直接折腾正式数据。

### 一个更安全的理解顺序

小白阶段不要一上来就在正式服务器上乱试。

更安全的顺序是：

1. 备份当前 SQLite 和 `media/`
2. 准备一个新的 PostgreSQL 空库
3. 设置 `DATABASE_URL` 指向 PostgreSQL
4. 执行 `migrate` 创建表结构
5. 再考虑从 SQLite 导出数据并导入
6. 检查页面、后台、图片

### 为什么 `media/` 仍然要保留

即使数据库迁到了 PostgreSQL，图片文件本体仍然不在数据库里。

所以：

```text
SQLite -> PostgreSQL
```

只是在迁移数据库内容。

你仍然要保证：

```text
media/ 文件还在，并且路径对应得上。
```

### 这一步不建议和前面部署链路混在一起排错

如果你同时做这些事：

- 第一次配 Nginx
- 第一次配 Gunicorn
- 第一次配 HTTPS
- 第一次迁 PostgreSQL

出问题会非常难判断。

所以更稳的做法是：

```text
先用 SQLite 让网站在服务器通过 IP 跑通
再单独做 PostgreSQL 迁移
```

### 这一章的检查问题

学完这一章，你应该能回答：

1. `DATABASE_URL` 是什么？
2. `postgres://用户名:密码@主机:端口/数据库名` 这串大概怎么拆？
3. 为什么数据库密码不应该写死在代码里？
4. `migrate` 是搬数据，还是创建/更新表结构？
5. `dumpdata`、`loaddata`、`migrate` 三者角色有什么区别？
6. 为什么迁 PostgreSQL 时仍然要关心 `media/`？

### 这一章的结论

这一章只要记住六句话：

1. SQLite 到 PostgreSQL 不是复制文件，而是换数据库服务。
2. `DATABASE_URL` 告诉 Django 生产环境连接哪个数据库。
3. `migrate` 主要负责创建或更新表结构，不会自动把 SQLite 内容搬到 PostgreSQL。
4. 迁已有内容通常需要导出数据、切换数据库、建表、再导入数据。
5. Wagtail 数据关系复杂，迁移前必须备份 SQLite 和 `media/`。
6. 最稳妥的路线是先用 SQLite 跑通部署，再单独迁 PostgreSQL。

## 16. 理解开发环境改代码后服务器为什么不会自动变

### 这一章要解决什么问题

前面已经讲了怎么让服务器运行网站。

接下来要解决另一个实际问题：

```text
如果我在本地开发环境改了代码，服务器上的网站会不会自动跟着变？
```

答案是：

```text
不会。
```

除非你把改动同步到服务器，并让服务器重新加载相关内容。

### 本地代码和服务器代码是两份东西

你本地电脑上有一份项目代码。

服务器上也会有一份项目代码。

它们可能来自同一个 Git 仓库，但它们仍然是两份不同机器上的文件。

所以：

```text
你本地改了 templates/
服务器上的 templates/ 不会自动变化。
```

同理：

```text
你本地改了 labsite/
服务器上的 labsite/ 也不会自动变化。
```

### 为什么这件事很重要

如果你没理解这一点，就会出现这样的困惑：

```text
我明明改了代码，为什么线上网站没变？
```

原因通常很简单：

```text
你只改了本地，还没有把改动发布到服务器。
```

### 本地开发和服务器运行的分工

可以这样理解：

| 环境 | 主要作用 |
| --- | --- |
| 本地开发环境 | 写代码、调试、验证效果 |
| 服务器生产环境 | 跑正式网站，让别人访问 |

所以本地改完代码后，还要有一个动作：

```text
发布到服务器
```

### 什么叫“发布”

对这个项目来说，“发布”不是发朋友圈，也不是 Wagtail 后台里的 Publish。

这里说的发布是：

```text
把本地代码改动同步到服务器，并让服务器使用新代码运行。
```

这个动作通常包括：

- 把代码更新到服务器
- 如果依赖变了，安装依赖
- 如果模型变了，执行数据库迁移
- 如果前端变了，重新构建静态资源
- 重启 Gunicorn 或相关服务

### 为什么只是复制文件不是好办法

小项目临时可以复制文件，但长期维护不推荐。

因为手工复制很容易：

- 漏掉文件
- 覆盖错文件
- 不知道服务器现在到底是哪一版
- 多人协作时产生混乱

更规范的方式是用 Git。

### Git 在这里的角色

Git 可以理解成：

```text
记录代码版本，并让本地和服务器都能拿到同一份版本记录。
```

标准思路是：

```text
本地改代码
  -> 提交到 Git
    -> 推送到远程仓库
      -> 服务器从远程仓库拉取更新
```

下一章会专门讲这个流程。

### 代码、数据、上传文件不要混为一谈

这里还要注意：

```text
代码更新
```

和：

```text
数据库内容更新
```

不是一回事。

比如你在本地改模板，这是代码。

但你在 Wagtail 后台发新闻，这是数据库内容。

再比如你上传图片，这是 `media/` 文件。

所以以后维护时要区分：

| 改动类型 | 主要影响 |
| --- | --- |
| 改模板、模型、样式、JS | 代码发布 |
| 后台发新闻、改页面文字 | 数据库内容 |
| 后台上传图片、附件 | `media/` |

### 这一章的检查问题

学完这一章，你应该能回答：

1. 本地代码改了，服务器代码会不会自动变？
2. 为什么线上网站没变，不一定是代码没写对？
3. “发布到服务器”大概包含哪些动作？
4. 为什么长期不推荐手工复制文件更新服务器？
5. 代码、数据库内容、`media/` 上传文件有什么区别？

### 这一章的结论

这一章只要记住五句话：

1. 本地电脑和服务器上通常是两份不同的项目文件。
2. 本地改代码不会自动影响服务器。
3. 让服务器变化，需要把代码发布到服务器，并让服务重新加载。
4. 长期维护不推荐手工复制文件，推荐使用 Git 流程。
5. 代码更新、数据库内容更新、`media/` 上传文件更新是三类不同事情。

## 17. 建立 Git 更新流程：本地提交，服务器拉取

### 这一章要解决什么问题

前一步已经知道：

```text
本地改代码，服务器不会自动变。
```

这一章要解决的是：

```text
那以后我本地改完代码，应该怎样让服务器更新？
```

这里先讲最基础、最常见的一条路线：

```text
本地提交 -> 推送远程仓库 -> 服务器拉取
```

### Git 在部署里的作用

Git 在这里不是为了炫技。

它的核心作用是：

```text
让本地和服务器都能基于同一个代码版本记录来同步。
```

这样你能知道：

- 本地改了什么
- 服务器当前是什么版本
- 哪次更新导致了问题
- 必要时如何回退到旧版本

### 基本流程长什么样

标准流程可以先记成：

```text
本地电脑
  -> git add
  -> git commit
  -> git push
  -> 服务器 git pull
```

翻译成人话就是：

```text
本地把改动记录下来
  -> 推到远程仓库
    -> 服务器从远程仓库取最新代码
```

### 本地通常做什么

在本地开发环境里，你改完代码后通常做：

```bash
git status
git add .
git commit -m "描述这次改动"
git push
```

这几条命令大概意思是：

| 命令 | 作用 |
| --- | --- |
| `git status` | 看哪些文件改了 |
| `git add .` | 把改动加入这次提交 |
| `git commit -m "..."` | 生成一次代码版本记录 |
| `git push` | 推送到远程仓库 |

### 服务器通常做什么

服务器上不应该随便手工改代码。

服务器通常只做：

```bash
cd /srv/lab-web
git pull
```

这表示：

```text
从远程仓库拉取最新代码到服务器。
```

### 为什么服务器上不建议直接改代码

如果你在服务器上直接改模板、改 Python 文件，会带来很多麻烦：

- 本地没有这份改动
- 远程仓库没有这份改动
- 下次 `git pull` 可能冲突
- 你很难知道线上现在到底改了什么

所以建议：

```text
本地负责改代码。
服务器负责运行代码。
```

### 哪些东西不应该靠 Git 同步

很重要：不是所有东西都应该进 Git。

通常不应该进 Git 的有：

- `.venv/`
- `node_modules/`
- `db.sqlite3`
- `media/`
- `.env`
- `static/`

你当前项目的 `.gitignore` 也已经忽略了其中一些，比如：

- `/.venv/`
- `*.sqlite3`
- `/static/`
- `node_modules/`

这说明项目本身就倾向于：

```text
代码走 Git；
运行环境、数据库、上传文件不要直接走 Git。
```

### Git 更新代码后，网站会立刻变吗

不一定。

`git pull` 只是让服务器磁盘上的代码文件更新了。

但如果网站进程还在用旧代码，或者前端静态文件还没重新构建，你还需要后续发布动作。

这些动作包括：

- 重启服务
- 执行 migration
- 构建前端
- collectstatic

下一章会专门讲服务器更新代码后的固定发布流程。

### 这一章的检查问题

学完这一章，你应该能回答：

1. 为什么本地改代码后要经过 Git 才适合更新服务器？
2. `git add`、`git commit`、`git push`、`git pull` 大概分别做什么？
3. 为什么不建议长期直接在服务器上改代码？
4. 哪些东西通常不应该进 Git？
5. 为什么 `git pull` 后网站不一定已经完整更新？

### 这一章的结论

这一章只要记住五句话：

1. 推荐的代码更新路线是：本地提交、推送远程、服务器拉取。
2. Git 负责同步代码版本，不负责同步数据库和上传文件。
3. 服务器上不建议直接改代码，否则容易和本地、仓库产生冲突。
4. `.venv`、`db.sqlite3`、`media/`、`node_modules/` 这类东西通常不靠 Git 同步。
5. `git pull` 只是更新代码文件，后面通常还要执行发布流程。

## 18. 服务器更新代码后的固定发布流程：依赖、迁移、构建、重启

### 这一章要解决什么问题

前一步讲了：

```text
服务器通过 git pull 拿到最新代码。
```

这一章要解决的是：

```text
服务器拿到新代码后，还要做哪些动作，网站才算真正更新完成？
```

### 先说结论

`git pull` 只是更新了服务器上的代码文件。

但一个 Django/Wagtail 项目更新后，可能还需要：

- 安装新的 Python 依赖
- 安装新的前端依赖
- 执行数据库迁移
- 构建前端静态资源
- 收集静态文件
- 重启 Gunicorn

这些合起来才叫比较完整的发布流程。

### 一个完整但偏保守的发布流程

小白阶段可以先记这个完整版本：

```bash
cd /srv/lab-web
git pull
.venv/bin/pip install -r requirements.txt
npm install
npm run build:prod
DJANGO_SETTINGS_MODULE=labsite.settings.production .venv/bin/python manage.py migrate
DJANGO_SETTINGS_MODULE=labsite.settings.production .venv/bin/python manage.py collectstatic --noinput
sudo systemctl restart lab-web
```

这不是说每次都绝对必须全部执行。

而是：

```text
刚开始不熟时，完整流程更不容易漏步骤。
```

### `pip install -r requirements.txt` 是干什么

```bash
.venv/bin/pip install -r requirements.txt
```

作用是：

```text
根据 requirements.txt 安装或更新 Python 依赖。
```

如果你这次更新没有改 Python 依赖，它可能不会做太多事。

但如果新增了依赖，而你忘了跑这一步，服务器可能报：

```text
ModuleNotFoundError
```

### `npm install` 是干什么

```bash
npm install
```

作用是：

```text
根据 package.json 安装或更新前端构建依赖。
```

如果前端依赖没有变化，它也可能不会做太多事。

但如果 `package.json` 变了，服务器没有安装新依赖，后面的前端构建可能失败。

### `npm run build:prod` 是干什么

```bash
npm run build:prod
```

作用是：

```text
把 static_src/ 里的前端源码编译成生产用静态资源。
```

如果你改了：

- Sass
- JavaScript
- Tailwind class
- 前端入口文件

通常都需要跑这一步。

### `migrate` 是干什么

```bash
DJANGO_SETTINGS_MODULE=labsite.settings.production .venv/bin/python manage.py migrate
```

这一步再次强调：

```text
migrate 不是把 SQLite 数据搬到 PostgreSQL。
```

在发布流程里，它的作用是：

```text
把代码里的数据库结构变更应用到当前生产数据库。
```

例如你改了模型：

- 新增字段
- 删除字段
- 修改字段属性

并且生成了 Django migration 文件，那么生产数据库也要执行：

```bash
manage.py migrate
```

这样数据库结构才能跟上代码。

### `collectstatic` 是干什么

```bash
DJANGO_SETTINGS_MODULE=labsite.settings.production .venv/bin/python manage.py collectstatic --noinput
```

作用是：

```text
把 Django/Wagtail/项目自身的静态文件统一收集到生产静态目录。
```

这一步通常放在前端构建之后，因为它要收集最新构建出来的静态产物。

### `systemctl restart lab-web` 是干什么

```bash
sudo systemctl restart lab-web
```

作用是：

```text
重启服务器上托管 Gunicorn 的服务，让它加载新代码。
```

如果你只 `git pull`，但不重启服务，正在运行的 Gunicorn 进程可能仍然使用旧代码。

所以代码更新后，通常需要重启应用服务。

### 哪些改动对应哪些发布动作

可以粗略这样记：

| 你改了什么 | 通常需要做什么 |
| --- | --- |
| Python 代码 | `git pull` + 重启服务 |
| 模板 HTML | `git pull` + 重启服务 |
| Sass / JS / Tailwind | 前端构建 + `collectstatic` + 重启服务 |
| Django 模型 | `migrate` + 重启服务 |
| Python 依赖 | `pip install` + 重启服务 |
| 前端依赖 | `npm install` + 前端构建 |

小白阶段不用急着精简。

先用完整流程，熟悉后再判断哪些步骤可以省。

### 为什么发布流程要固定

固定流程的好处是：

- 不容易漏步骤
- 出问题容易排查
- 以后别人接手也知道怎么操作
- 你自己过几个月回来也不会忘

部署最怕的是：

```text
每次凭感觉敲几条命令。
```

长期一定要形成固定流程。

### 这一章的检查问题

学完这一章，你应该能回答：

1. 为什么 `git pull` 后网站不一定完整更新？
2. `pip install -r requirements.txt` 是解决什么问题？
3. `npm run build:prod` 和 `collectstatic` 的位置为什么通常是先后关系？
4. 发布流程里的 `migrate` 是干什么，不是干什么？
5. 为什么代码更新后通常要重启 Gunicorn 服务？

### 这一章的结论

这一章只要记住六句话：

1. `git pull` 只是更新代码文件，不等于完整发布完成。
2. Python 依赖变化要跑 `pip install`。
3. 前端依赖和源码变化要跑 `npm install` 和 `npm run build:prod`。
4. 数据库结构变化要跑 `migrate`，但它不是自动搬 SQLite 数据。
5. 静态资源生产环境要跑 `collectstatic`。
6. 最后要重启应用服务，让 Gunicorn 加载新代码。

## 19. 理解域名、DNS、A 记录和服务器 IP 的关系

### 这一章要解决什么问题

前面一直在讲：

```text
通过服务器 IP 访问网站。
```

这一章开始进入：

```text
怎么让别人通过域名访问网站。
```

这一步先只讲概念：

```text
域名、DNS、A 记录、服务器 IP 之间是什么关系？
```

### IP 和域名的区别

IP 是服务器在网络里的地址，例如：

```text
1.2.3.4
```

域名是人类更容易记住的名字，例如：

```text
lab.example.com
```

浏览器最终访问服务器时，仍然需要找到 IP。

所以域名背后要有一套机制，把：

```text
lab.example.com
```

翻译成：

```text
1.2.3.4
```

这套机制就是 DNS。

### DNS 是什么

DNS 可以先理解成：

```text
互联网的电话簿。
```

你告诉它一个域名，它告诉你这个域名对应哪个 IP。

所以浏览器访问：

```text
http://lab.example.com
```

大致会先问：

```text
lab.example.com 对应哪个 IP？
```

DNS 回答：

```text
它对应 1.2.3.4。
```

然后浏览器再去访问这台服务器。

### A 记录是什么

A 记录是 DNS 里最常见的一种记录。

它的作用是：

```text
把一个域名指向一个 IPv4 地址。
```

例如：

| 类型 | 主机记录 | 记录值 |
| --- | --- | --- |
| A | `lab` | `1.2.3.4` |

如果你的主域名是：

```text
example.com
```

那么 `lab` 这个主机记录通常表示：

```text
lab.example.com
```

所以这条 A 记录的意思是：

```text
lab.example.com 指向 1.2.3.4。
```

### 域名解析完成后，网站就一定能访问吗

不一定。

域名解析只解决：

```text
域名 -> 服务器 IP
```

它不负责：

- 服务器有没有开机
- Nginx 有没有配置这个域名
- Django 的 `ALLOWED_HOSTS` 是否允许这个域名
- HTTPS 证书是否配置好

所以域名绑定要分两层理解：

1. DNS 层

   ```text
   让域名找到服务器 IP。
   ```

2. 服务器层

   ```text
   让 Nginx 和 Django 知道这个域名应该由当前网站处理。
   ```

下一章才会讲服务器层怎么配。

### 为什么 DNS 可能不是立刻生效

你在域名服务商后台改了 A 记录后，有时不是全世界立刻都能访问新地址。

因为 DNS 有缓存。

所以你可能会遇到：

- 你电脑能访问
- 别人电脑还不行
- 或者过一会儿才生效

这不一定是服务器坏了，可能只是 DNS 传播和缓存还没完全更新。

### 这一章的检查问题

学完这一章，你应该能回答：

1. IP 和域名分别是什么？
2. DNS 的作用是什么？
3. A 记录是干什么的？
4. `lab.example.com A 1.2.3.4` 这句话是什么意思？
5. 为什么域名解析到服务器 IP 后，网站仍然可能打不开？

### 这一章的结论

这一章只要记住五句话：

1. IP 是服务器地址，域名是更容易记住的名字。
2. DNS 负责把域名解析成 IP。
3. A 记录负责把域名指向一个 IPv4 地址。
4. 域名解析只解决“找到哪台服务器”，不解决服务器内部配置。
5. 域名真正可用，还需要 Nginx、Django 和 HTTPS 配合配置。

## 20. 把域名解析到服务器 IP，并配置 Nginx `server_name`

### 这一章要解决什么问题

前一步讲了：

```text
DNS 负责把域名解析到服务器 IP。
```

这一章要解决的是：

```text
域名已经指到服务器以后，服务器里面还要怎么认这个域名？
```

这一步主要涉及三处：

- DNS A 记录
- Nginx `server_name`
- Django / Wagtail 相关环境变量

### 第一步：DNS 里加 A 记录

在域名服务商后台添加类似记录：

| 类型 | 主机记录 | 记录值 |
| --- | --- | --- |
| A | `lab` | `服务器公网 IP` |

如果主域名是：

```text
example.com
```

那么这条记录表示：

```text
lab.example.com -> 服务器公网 IP
```

这一步完成后，浏览器访问：

```text
http://lab.example.com
```

理论上会来到你的服务器。

### 第二步：Nginx 里配置 `server_name`

DNS 只负责把请求带到服务器。

到了服务器以后，Nginx 还要知道：

```text
这个域名应该交给哪个站点配置处理。
```

这就是 `server_name` 的作用。

例如：

```nginx
server {
    listen 80;
    server_name lab.example.com;

    location / {
        proxy_pass http://127.0.0.1:8000;
        proxy_set_header Host $host;
        proxy_set_header X-Real-IP $remote_addr;
        proxy_set_header X-Forwarded-For $proxy_add_x_forwarded_for;
        proxy_set_header X-Forwarded-Proto $scheme;
    }
}
```

这里的核心变化是：

```nginx
server_name lab.example.com;
```

意思是：

```text
当请求的 Host 是 lab.example.com 时，用这个 server 块处理。
```

### 第三步：Django 允许这个域名

Django 还有一个安全检查：

```text
ALLOWED_HOSTS
```

它的作用是：

```text
限制哪些 Host 可以访问这个 Django 项目。
```

生产环境应该设置类似：

```bash
ALLOWED_HOSTS=lab.example.com
```

如果有多个，可以逗号分隔，例如：

```bash
ALLOWED_HOSTS=lab.example.com,www.example.com
```

如果 Nginx 配好了，但 Django 不允许这个域名，可能会出现：

```text
DisallowedHost
```

之类的问题。

### 第四步：设置 `CSRF_TRUSTED_ORIGINS`

Django 还会检查 CSRF 相关来源。

生产域名需要设置：

```bash
CSRF_TRUSTED_ORIGINS=https://lab.example.com
```

注意这里要带：

```text
https://
```

不是只写域名。

如果暂时还没配 HTTPS，只是在 HTTP 阶段测试，概念上也要知道正式环境最终应该写 HTTPS 地址。

### 第五步：设置 `WAGTAILADMIN_BASE_URL`

Wagtail 后台还需要知道站点的后台基础 URL。

建议设置：

```bash
WAGTAILADMIN_BASE_URL=https://lab.example.com
```

它会影响后台里一些完整链接、通知链接等场景。

你这个项目当前默认值还是示例地址，所以正式上线前必须改成真实域名。

### 这几层分别负责什么

| 层 | 配置 | 作用 |
| --- | --- | --- |
| DNS | A 记录 | 让域名找到服务器 IP |
| Nginx | `server_name` | 让服务器知道这个域名交给哪个站点处理 |
| Django | `ALLOWED_HOSTS` | 允许这个 Host 访问应用 |
| Django | `CSRF_TRUSTED_ORIGINS` | 信任这个域名来源 |
| Wagtail | `WAGTAILADMIN_BASE_URL` | 后台使用正确站点基础 URL |

### 为什么域名解析成功但仍可能打不开

如果 DNS 已经生效，但网站打不开，常见原因是：

- Nginx 没配置这个 `server_name`
- Nginx 配置改了但没 reload
- Django 的 `ALLOWED_HOSTS` 没包含这个域名
- 生产环境变量没生效
- Gunicorn 没重启

所以排查时要按层看，不要只盯着 DNS。

### 这一章的检查问题

学完这一章，你应该能回答：

1. DNS A 记录解决什么问题？
2. Nginx `server_name` 解决什么问题？
3. Django `ALLOWED_HOSTS` 是干什么的？
4. `CSRF_TRUSTED_ORIGINS` 为什么要带 `https://`？
5. `WAGTAILADMIN_BASE_URL` 为什么要改成正式域名？

### 这一章的结论

这一章只要记住五句话：

1. DNS A 记录只负责把域名指向服务器 IP。
2. Nginx `server_name` 负责让服务器知道这个域名该由哪个站点处理。
3. Django `ALLOWED_HOSTS` 负责允许这个域名访问应用。
4. `CSRF_TRUSTED_ORIGINS` 和 `WAGTAILADMIN_BASE_URL` 也要改成正式域名。
5. 域名可用不是单点配置，而是 DNS、Nginx、Django/Wagtail 多层配合。

## 21. 配置 HTTPS，让域名用安全访问

### 这一章要解决什么问题

前一步已经讲到：

```text
域名可以解析到服务器，并由 Nginx 和 Django 识别。
```

这一章要解决的是：

```text
为什么正式站点还必须配 HTTPS？它和当前项目配置有什么关系？
```

### HTTPS 是什么

可以先用最简单的话理解：

```text
HTTPS 就是加密的网页访问。
```

如果只有 HTTP，浏览器和服务器之间的数据传输更容易被窃听或篡改。

而 HTTPS 会让：

- 浏览器和服务器之间的通信加密
- 浏览器显示安全锁标志
- 后台登录、Cookie 等更安全

### 为什么正式网站不能长期停留在纯 HTTP

因为正式网站不仅有公开页面，还有：

- 后台登录
- 管理员 Cookie
- 表单提交
- 上传文件

这些都不适合长期裸奔在纯 HTTP 上。

尤其是 Wagtail 后台这种管理入口，更应该跑在 HTTPS 下。

### 为什么对你这个项目尤其重要

你这个项目的生产配置里已经有这些设置：

```text
SESSION_COOKIE_SECURE = True
CSRF_COOKIE_SECURE = True
SECURE_SSL_REDIRECT = True
```

它们大概分别表示：

| 配置 | 大致作用 |
| --- | --- |
| `SESSION_COOKIE_SECURE = True` | 登录会话 Cookie 只通过 HTTPS 发送 |
| `CSRF_COOKIE_SECURE = True` | CSRF Cookie 只通过 HTTPS 发送 |
| `SECURE_SSL_REDIRECT = True` | 强制把 HTTP 跳转到 HTTPS |

这说明：

```text
这个项目的生产配置本来就假设你最终会用 HTTPS。
```

### HTTPS 依赖什么

要让 HTTPS 工作，光有域名还不够，还要有：

1. 域名已经正确解析到服务器
2. Nginx 已经能处理这个域名
3. 服务器有 HTTPS 证书

这个证书可以理解成：

```text
证明“这个域名确实属于这台服务器”的数字凭证。
```

### 常见做法：Let's Encrypt + Certbot

现在最常见的免费方案是：

- Let's Encrypt 提供证书
- Certbot 帮你申请和配置证书

常见命令类似：

```bash
sudo apt install -y certbot python3-certbot-nginx
sudo certbot --nginx -d lab.example.com
```

小白阶段先不用死记命令细节，先理解它在做什么：

```text
帮你的域名申请 HTTPS 证书，并自动改 Nginx 配置。
```

### HTTPS 配好后，访问链路会怎样变化

原来：

```text
浏览器
  -> http://lab.example.com
    -> Nginx
      -> Gunicorn
```

配好后，正式目标会变成：

```text
浏览器
  -> https://lab.example.com
    -> Nginx
      -> Gunicorn
```

并且通常希望：

```text
http://lab.example.com
```

自动跳转到：

```text
https://lab.example.com
```

### 为什么 `X-Forwarded-Proto` 在这里很重要

前面 Nginx 配置里有：

```nginx
proxy_set_header X-Forwarded-Proto $scheme;
```

而 Django 里有：

```text
SECURE_PROXY_SSL_HEADER = ('HTTP_X_FORWARDED_PROTO', 'https')
```

这两者配合的作用是：

```text
虽然 Django 在后面看到的是 Nginx 转发过来的请求，
但它仍然能知道用户最外层访问时是不是 HTTPS。
```

这对安全配置很重要。

### HTTPS 配好后要检查什么

至少检查：

1. `https://你的域名` 能打开
2. 浏览器没有明显“不安全”提示
3. `http://你的域名` 会跳转到 `https://你的域名`
4. `/admin/` 能正常登录
5. 上传图片和表单提交正常

### 这一章的检查问题

学完这一章，你应该能回答：

1. HTTPS 和 HTTP 的核心区别是什么？
2. 为什么正式 Wagtail 后台更应该使用 HTTPS？
3. 你这个项目哪些生产配置说明它本来就假设 HTTPS 会开启？
4. 域名可用后，为什么还需要证书才能上 HTTPS？
5. 为什么 HTTP 到 HTTPS 的跳转是正式站点常见要求？

### 这一章的结论

这一章只要记住五句话：

1. HTTPS 是加密访问，正式网站尤其是后台管理页面应当使用 HTTPS。
2. 你这个项目的生产配置已经明显按 HTTPS 场景设计。
3. 配 HTTPS 不只要域名，还要证书和正确的 Nginx 配置。
4. 常见做法是用 Let's Encrypt 和 Certbot 申请并配置证书。
5. 正式上线时通常要让 HTTP 自动跳转到 HTTPS。

## 22. 建立备份和恢复习惯：数据库、`media/`、代码版本

### 这一章要解决什么问题

前面已经把“怎么部署上线”走完了。

最后一步要解决的是：

```text
网站上线后，出了问题怎么办？服务器坏了怎么办？误操作了怎么办？
```

真正长期维护网站，最怕的通常不是部署命令难，而是：

```text
数据丢了，恢复不了。
```

所以最后这一步讲的是：

```text
要备份什么，以及为什么“恢复能力”比“有个压缩包”更重要。
```

### 至少要备份哪三类东西

对你这个项目，至少要考虑这三类：

1. 数据库
2. `media/`
3. 代码版本

可以先记成：

| 类别 | 负责什么 |
| --- | --- |
| 数据库 | 页面内容、用户、权限、结构记录 |
| `media/` | 上传图片和附件文件本体 |
| 代码版本 | 模板、逻辑、样式、模型定义 |

### 为什么只备份数据库不够

如果只备份数据库，没有备份 `media/`，会发生：

- 页面内容可能还能恢复
- 但图片、附件可能全没了

也就是：

```text
文字还在，文件丢了。
```

### 为什么只备份 `media/` 也不够

如果只备份 `media/`，没有数据库，会发生：

- 服务器上有很多图片文件
- 但 Wagtail 不知道这些文件属于谁

也就是：

```text
文件还在，记录丢了。
```

### 为什么代码版本也要备份

很多人会以为代码在 Git 仓库里，所以不用管了。

实际上代码版本当然最容易恢复，但也仍然应该明确：

- 正式网站当前跑的是哪个提交
- 生产环境用的配置是什么
- Nginx 配置和 systemd 配置在哪

所以严格一点说，除了 Git 仓库，还应该保留：

- 生产部署说明
- Nginx 配置
- systemd 服务配置
- 环境变量记录方式

### 什么叫“恢复能力比备份文件更重要”

很多人说“我有备份”，其实只是：

```text
某个目录里躺着一个压缩包。
```

但真正重要的问题是：

```text
如果今天服务器坏了，你能不能真的把网站恢复出来？
```

所以比“有没有备份文件”更重要的是：

```text
你知不知道怎么把它恢复回来。
```

### 最低要求的备份思路

小白阶段，先做到最低要求就已经比很多人强了：

1. 定期备份数据库
2. 定期备份 `media/`
3. 代码持续提交到 Git
4. 备份不要只放在服务器自己机器上

因为如果备份也放在同一台服务器上，而服务器整个磁盘坏了，等于还是一起没了。

### 备份放哪比较合理

常见选择有：

- 另一台服务器
- 对象存储
- 本地电脑
- 学校或单位提供的备份空间

关键原则是：

```text
不要只备份在和生产站点同一块磁盘上。
```

### 恢复演练为什么重要

最稳妥的做法不是“盲相信备份存在”，而是：

```text
至少做过一次恢复演练。
```

例如你要知道：

- 数据库备份如何恢复
- `media/` 如何放回正确目录
- 代码如何拉回正确版本
- Nginx 和 Gunicorn 如何重新启动

哪怕只做过一次，你以后心里也会踏实很多。

### 这一章的检查问题

学完这一章，你应该能回答：

1. 对这个项目来说，至少要备份哪三类东西？
2. 为什么只备份数据库不够？
3. 为什么只备份 `media/` 也不够？
4. 为什么说“能恢复”比“有备份文件”更重要？
5. 为什么备份不应该只放在生产服务器本机？

### 这一章的结论

这一章只要记住六句话：

1. 这个项目至少要同时备份数据库、`media/` 和代码版本。
2. 只备份数据库会丢图片和附件，只备份 `media/` 会丢内容记录。
3. Git 很重要，但生产配置和部署信息也要能找回。
4. 备份不能只放在生产服务器本机。
5. 真正重要的不只是“有备份”，而是“你会恢复”。
6. 上线完成不代表结束，长期维护从备份习惯开始。
