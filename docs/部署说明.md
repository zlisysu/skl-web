# 部署说明

本文记录本项目在 Linux 本机/服务器上的部署方式。当前推荐方案是：

```text
浏览器
  -> Nginx :80 或 :443
    -> /static/ 由 Nginx 直接读取 static/
    -> /media/ 由 Nginx 直接读取 media/
    -> 其他请求转发到 Gunicorn 127.0.0.1:8000
      -> Django/Wagtail
        -> db.sqlite3
```

当前阶段可以继续使用 `db.sqlite3` 和 `media/` 文件夹。等访问量、多人后台编辑、备份要求上来后，再迁到 PostgreSQL 或 SQL Server。

## 1. 部署目录

不要把正式部署目录放在用户 home 目录下，例如：

```text
/home/zli/test/skl-web
```

原因是 Nginx 默认以 `www-data` 用户运行，而用户 home 目录通常权限较收，例如 `/home/zli` 可能是 `750`，导致 Nginx 进不去，表现为页面能打开但 CSS、图片、JS 加载失败。

推荐放到：

```text
/srv/skl-web
```

新环境上线时，建议直接从 GitHub 拉代码到 `/srv/skl-web`：

```bash
sudo mkdir -p /srv/skl-web
sudo chown zli:www-data /srv/skl-web
git clone git@github.com:zlisysu/skl-web.git /srv/skl-web
sudo chmod -R g+rX /srv/skl-web
```

后续更新代码时：

```bash
cd /srv/skl-web
git pull
```

注意：Git 仓库不包含正式数据文件。继续使用 SQLite 时，`db.sqlite3` 和 `media/` 要从旧环境单独复制到 `/srv/skl-web`。复制后再修正写权限：

```bash
sudo chown -R zli:www-data /srv/skl-web/db.sqlite3 /srv/skl-web/media
sudo chmod -R g+w /srv/skl-web/media
sudo chmod g+w /srv/skl-web/db.sqlite3
```

## 2. Python 环境

如果系统有 `python3.12-venv`：

```bash
cd /srv/skl-web
python3.12 -m venv .venv
.venv/bin/pip install -r requirements.txt
```

如果创建 venv 时报错缺少 `ensurepip`，需要安装系统包：

```bash
sudo apt update
sudo apt install python3.12-venv python3.12-dev build-essential
```

如果没有 sudo，但有 conda，可以用 conda 环境替代 `.venv`：

```bash
conda create -n skl-web python=3.12 -y
conda activate skl-web
pip install -r requirements.txt
```

systemd 的 `ExecStart` 要使用实际的 `gunicorn` 路径。`.venv` 环境通常是：

```text
/srv/skl-web/.venv/bin/gunicorn
```

conda 环境可用下面命令查看：

```bash
which gunicorn
```

## 3. 前端与静态文件

前端源码在 `static_src/`，Django/Nginx 实际使用的静态文件在 `static/`。

部署时执行：

```bash
cd /srv/skl-web
npm ci
npm run build:prod
.venv/bin/python manage.py collectstatic --noinput
```

如果使用 conda 环境，把 `.venv/bin/python` 换成 `python`。

验证：

```bash
ls /srv/skl-web/static/css/main.css
```

## 4. 环境变量

生产服务使用 `/etc/skl-web.env` 保存环境变量：

```bash
sudo vi /etc/skl-web.env
```

本机 HTTP 测试可写：

```ini
DJANGO_SETTINGS_MODULE=labsite.settings.production
SECRET_KEY=换成强随机字符串
ALLOWED_HOSTS=127.0.0.1,localhost,192.168.1.20
CSRF_TRUSTED_ORIGINS=http://127.0.0.1,http://localhost,http://192.168.1.20,http://127.0.0.1:8000,http://localhost:8000,http://192.168.1.20:8000
```

说明：

- `DJANGO_SETTINGS_MODULE`：指定使用生产配置 `labsite.settings.production`。
- `SECRET_KEY`：Django 加密密钥，必须是长随机字符串，不能用示例值。
- `ALLOWED_HOSTS`：允许访问网站的主机名或 IP，不带 `http://`。
- `CSRF_TRUSTED_ORIGINS`：允许提交表单的来源，必须带 `http://` 或 `https://`。

生成 `SECRET_KEY`：

```bash
cd /srv/skl-web
.venv/bin/python -c "from django.core.management.utils import get_random_secret_key; print(get_random_secret_key())"
```

如果正式使用 HTTPS 和域名，例如 `skl.example.edu.cn`：

```ini
ALLOWED_HOSTS=skl.example.edu.cn
CSRF_TRUSTED_ORIGINS=https://skl.example.edu.cn
```

如果继续用项目目录下的 `db.sqlite3` 和 `media/`，不需要设置 `DATABASE_URL` 和 `MEDIA_ROOT`。项目默认会使用：

```text
/srv/skl-web/db.sqlite3
/srv/skl-web/media/
```

## 5. Django 初始化检查

```bash
cd /srv/skl-web
.venv/bin/python manage.py migrate
.venv/bin/python manage.py check --deploy
```

如果还没有配置 HTTPS，本机 HTTP 测试时需要临时关闭生产配置里的 HTTPS 强制项：

```python
# labsite/settings/production.py
SECURE_SSL_REDIRECT = False
SESSION_COOKIE_SECURE = False
CSRF_COOKIE_SECURE = False
```

正式 HTTPS 上线后再改回更安全的配置。

## 6. Gunicorn systemd 服务

创建服务文件：

```bash
sudo vi /etc/systemd/system/skl-web.service
```

使用端口转发方式，便于本机测试：

```ini
[Unit]
Description=skl-web gunicorn
After=network.target

[Service]
User=zli
Group=www-data
WorkingDirectory=/srv/skl-web
EnvironmentFile=/etc/skl-web.env
ExecStart=/srv/skl-web/.venv/bin/gunicorn labsite.wsgi:application \
    --workers 3 \
    --bind 127.0.0.1:8000 \
    --access-logfile - \
    --error-logfile -
Restart=always
RestartSec=3

[Install]
WantedBy=multi-user.target
```

启动：

```bash
sudo systemctl daemon-reload
sudo systemctl enable --now skl-web
sudo systemctl status skl-web --no-pager -l
```

验证 Gunicorn：

```bash
curl -I http://127.0.0.1:8000
```

如果返回 `301` 跳到 HTTPS，说明 `SECURE_SSL_REDIRECT=True` 仍在生效。HTTP 测试阶段按上一节临时关闭。

## 7. Nginx 配置

安装并启动 Nginx：

```bash
sudo apt update
sudo apt install nginx
sudo systemctl enable --now nginx
```

创建站点配置：

```bash
sudo vi /etc/nginx/sites-available/skl-web
```

HTTP 本机/局域网测试配置：

```nginx
server {
    listen 80;
    server_name _;

    client_max_body_size 50M;

    location /static/ {
        alias /srv/skl-web/static/;
    }

    location /media/ {
        alias /srv/skl-web/media/;
    }

    location / {
        proxy_pass http://127.0.0.1:8000;
        proxy_set_header Host $host;
        proxy_set_header X-Real-IP $remote_addr;
        proxy_set_header X-Forwarded-For $proxy_add_x_forwarded_for;
        proxy_set_header X-Forwarded-Proto $scheme;
    }
}
```

启用站点，并关闭默认站点，避免默认 Nginx 页面抢请求：

```bash
sudo ln -s /etc/nginx/sites-available/skl-web /etc/nginx/sites-enabled/skl-web
sudo rm -f /etc/nginx/sites-enabled/default
sudo nginx -t
sudo systemctl reload nginx
```

验证：

```bash
curl -I http://127.0.0.1
curl -I http://127.0.0.1/static/css/main.css
```

`/static/css/main.css` 应返回 `200 OK`。如果返回 `403 Forbidden`，通常是 Nginx 没有权限进入静态文件所在目录，优先确认项目是否放在 `/srv/skl-web`，以及权限是否给到 `www-data` 组。

## 8. 数据与媒体迁移

继续使用 SQLite 时，迁移到新机器至少带走：

```text
db.sqlite3
media/
```

示例：

```bash
scp db.sqlite3 user@server:/srv/skl-web/
rsync -av media/ user@server:/srv/skl-web/media/
```

迁移后修正权限：

```bash
sudo chown -R zli:www-data /srv/skl-web/db.sqlite3 /srv/skl-web/media
sudo chmod g+w /srv/skl-web/db.sqlite3
sudo chmod -R g+rwX /srv/skl-web/media
```

如果未来迁到 PostgreSQL 或 SQL Server，再使用：

```bash
.venv/bin/python manage.py dumpdata \
  --natural-foreign \
  --natural-primary \
  -e contenttypes \
  -e auth.Permission \
  --indent 2 > data.json
```

新数据库先 `migrate`，再 `loaddata data.json`。

## 9. 常见问题

### 页面能打开，但没有 CSS

优先检查：

```bash
curl -I http://127.0.0.1/static/css/main.css
namei -l /srv/skl-web/static/css/main.css
```

常见原因：

- 项目放在 `/home/zli/...`，Nginx 无权进入 `/home/zli`。
- Nginx `alias` 路径仍指向旧目录。
- 没有执行 `collectstatic`。
- 默认站点 `/etc/nginx/sites-enabled/default` 仍启用。

### DisallowedHost

把访问用的主机名或 IP 加到 `/etc/skl-web.env`：

```ini
ALLOWED_HOSTS=127.0.0.1,localhost,192.168.1.20
```

修改后重启：

```bash
sudo systemctl restart skl-web
```

### 后台登录或发布页面 CSRF 报错

把浏览器实际访问的来源加到 `CSRF_TRUSTED_ORIGINS`。注意必须带协议：

```ini
CSRF_TRUSTED_ORIGINS=http://127.0.0.1,http://localhost,http://192.168.1.20
```

修改后：

```bash
sudo systemctl restart skl-web
```

### 查看日志

```bash
sudo journalctl -u skl-web --no-pager -n 100
sudo tail -n 100 /var/log/nginx/error.log
```

## 10. 正式 HTTPS 上线

正式有域名和证书后：

1. Nginx `server_name` 改成正式域名。
2. `/etc/skl-web.env` 中：

```ini
ALLOWED_HOSTS=正式域名
CSRF_TRUSTED_ORIGINS=https://正式域名
```

3. `production.py` 中恢复：

```python
SECURE_SSL_REDIRECT = True
SESSION_COOKIE_SECURE = True
CSRF_COOKIE_SECURE = True
```

4. 确认：

```bash
sudo nginx -t
sudo systemctl reload nginx
sudo systemctl restart skl-web
.venv/bin/python manage.py check --deploy
```

## 11. 挂在学校网站子路径下

如果最终访问地址不是独立域名根路径，而是类似：

```text
https://sps.sysu.edu.cn/skl/
```

这属于子路径部署，路径前缀是：

```text
/skl
```

这种方式需要同时处理两件事：

- Nginx 把 `/skl/` 下的请求转发给本项目。
- Django/Wagtail 生成静态文件、媒体文件和后台链接时带上 `/skl` 前缀。

如果可以选择，优先使用独立子域名，例如：

```text
https://skl.sps.sysu.edu.cn/
```

独立子域名最省心，不需要路径前缀适配。只有学校最终只能提供 `https://sps.sysu.edu.cn/skl/` 时，才按下面方式配置。

### Django 环境变量

在 `/etc/skl-web.env` 中使用：

```ini
ALLOWED_HOSTS=sps.sysu.edu.cn
CSRF_TRUSTED_ORIGINS=https://sps.sysu.edu.cn
FORCE_SCRIPT_NAME=/skl
```

项目代码还需要在 `labsite/settings/base.py` 中支持 `FORCE_SCRIPT_NAME`。建议把当前固定的：

```python
STATIC_URL = "/static/"
MEDIA_URL = "/media/"
```

改成：

```python
FORCE_SCRIPT_NAME = os.environ.get("FORCE_SCRIPT_NAME") or None

if FORCE_SCRIPT_NAME:
    url_prefix = FORCE_SCRIPT_NAME.rstrip("/")
    STATIC_URL = url_prefix + "/static/"
    MEDIA_URL = url_prefix + "/media/"
else:
    STATIC_URL = "/static/"
    MEDIA_URL = "/media/"
```

否则页面会去请求：

```text
https://sps.sysu.edu.cn/static/css/main.css
```

而不是正确的：

```text
https://sps.sysu.edu.cn/skl/static/css/main.css
```

这会导致页面没有 CSS。

### Nginx 子路径配置

在 `sps.sysu.edu.cn` 对应的 Nginx server 中增加：

```nginx
location = /skl {
    return 301 /skl/;
}

location /skl/static/ {
    alias /srv/skl-web/static/;
}

location /skl/media/ {
    alias /srv/skl-web/media/;
}

location /skl/ {
    proxy_pass http://127.0.0.1:8000/;
    proxy_set_header Host $host;
    proxy_set_header X-Real-IP $remote_addr;
    proxy_set_header X-Forwarded-For $proxy_add_x_forwarded_for;
    proxy_set_header X-Forwarded-Proto $scheme;
}
```

注意 `proxy_pass` 后面有 `/`：

```nginx
proxy_pass http://127.0.0.1:8000/;
```

这表示 Nginx 会把外部的 `/skl/` 前缀剥掉，再转给 Django。

### Wagtail 后台

Wagtail 后台的 `Settings / Sites` 中，hostname 仍然填：

```text
sps.sysu.edu.cn
```

部署后重点测试：

```text
https://sps.sysu.edu.cn/skl/
https://sps.sysu.edu.cn/skl/admin/
https://sps.sysu.edu.cn/skl/static/css/main.css
```

如果后台登录、发布页面或图片上传出现跳转到根路径的问题，优先检查 `FORCE_SCRIPT_NAME` 是否已进入 systemd 环境，并重启 `skl-web` 服务。
